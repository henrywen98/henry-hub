# 概要设计说明书组装器

概要设计说明书是合同要求的交付物，核心读者是**甲方**（项目负责人、业务部门、技术评审），用于方案确认阶段让客户理解"我们打算怎么做"并签字确认。因此文档语言需要兼顾业务可读性和技术准确性——第1章背景和第3章功能描述要让非技术人员能看懂，技术章节（第2/5/7章）则面向双方技术团队。本 skill 将当前目录下的素材组装为标准 7 章概设文档。

## 为什么是"组装"不是"生成"

概设在需求确认单之后产出，功能范围已经确认过。概设的真正增量是把已确认的需求包装进标准技术方案外壳。大部分技术/安全/部署章节在项目间变化不大，可以从过往概设直接复用。

所以 skill 的工作是：**把需求确认单的功能列表转成概设格式 + 从过往概设复用其他章节 + 微调项目细节**。

## 输入素材（全部在当前目录下）

```
当前目录/
├── 需求确认单 / PRD           ← 必须有（第3章功能模块的来源）
├── 过往概设 / 概设模板         ← 最重要的参考（其他所有章节的骨架和内容来源）
├── 技术架构文档               ← 可选（有则用来细化第2/5章，没有就从过往概设复用）
└── 原型 / prototype           ← 可选（有则辅助第5.5章界面设计描述）
```

---

## 工作流程

```
Phase 1: 素材扫描 → 识别当前目录下的输入文件
Phase 2: 确认 → 确认文件 + 收集项目元信息
Phase 3: 内容提取 → 从素材中提取结构化内容
Phase 4: 组装输出 → 按模板合成概要设计说明书
Phase 5: 核验报告 → 自动验证 + 交付用户
```

## Phase 1: 素材扫描

使用 Glob 扫描当前目录下的文档文件（`.md`、`.docx`、`.doc`、`.pdf`），按以下规则识别：

### 必需素材

| 素材 | 识别规则 | 用途 |
|------|---------|------|
| 需求确认单 / PRD | 文件名含 `需求确认`/`确认单`/`PRD`/`prd`/`需求` | 第3章功能模块的来源 |

缺失则阻断，提示用户先准备需求文档。

### 关键参考

| 素材 | 识别规则 | 说明 |
|------|---------|------|
| 过往概设 / 概设模板 | 文件名含 `概设`/`概要设计`/`HLD`/`模板` | 第1/2/4/5/6/7章的骨架来源 |

缺失时提示用户：过往概设是最重要的参考素材，缺少它技术/安全等章节会使用内置模板（质量和匹配度会降低）。用户确认后继续。

### 可选素材

| 素材 | 识别规则 | 降级策略 |
|------|---------|---------|
| 技术架构文档 | 文件名含 `技术架构`/`架构设计`/`architecture` | 无则从过往概设复用技术章节 |
| 接口设计文档 | 文件名含 `接口`/`api`/`interface` | 无则第6章仅写概述 |
| 原型文件 | `prototype`/`原型` 目录或文件 | 无则第5.5章用通用描述 |

## Phase 2: 确认

1. **素材确认**：列出识别到的文件路径，请用户确认。用户可手动指定路径或补充遗漏文件。

2. **收集项目元信息**（带默认值）：
   - 项目全称
   - 项目简称（文档内引用）
   - 承建单位名称
   - 文档日期（默认当月）
   - 文档版本（默认 V1.0）

3. **快速确认**：问用户"技术架构和过往概设基本一致，还是有需要调整的地方？"如果一致，技术章节直接复用；如果有变化，记录变化点。

## Phase 3: 内容提取

使用 Agent 工具（`subagent_type: Explore`）并行提取。所有 Agent 在同一条消息中发出。

### Agent 切分策略

根据需求文档中的模块数 N（用 Grep 搜索模块级标题计算）：
- N ≤ 5：1 个模块 Agent
- 6 ≤ N ≤ 15：2 个模块 Agent
- N > 15：3 个模块 Agent，每个处理约 N/3

### Agent 定义

**Agent 1..K — 需求功能提取**（`subagent_type: Explore`）

```
读取文件 {requirements_path}，提取以下模块的功能描述：{module_list}

对每个模块输出：
1. 模块简述（2-3句话），需要包含：
   - 模块在整个平台中的定位（如"作为平台的核心业务模块"/"作为日常办公的支撑模块"）
   - 核心场景和使用角色
   - 与其他子系统的关联关系（如"与财务管理子系统联动实现付款审批"）
2. 功能设计表格，每个子功能一行：

| 序号 | 业务模块 | 功能名称 | 功能要求 |

"功能要求"列写作规范：
- 以"用于{业务场景描述}"开头
- 多子功能用"功能包括但不限于："引出
- 每个子功能以"支持"开头，分号分隔
- 量化用"不少于X个"，举例用"包括但不限于：{示例}等"
- 简单功能可只有用途说明

要求：
- 完整覆盖需求文档中该模块的所有功能点
- 第三人称客观描述
- 不写技术实现细节
```

**Agent K+1 — 过往概设提取**（`subagent_type: Explore`）

```
读取文件 {past_hld_path}，提取以下内容：

1. 第1章引言：编写目的、读者对象的完整文本
2. 第2章技术选型：硬件环境表格 + 体系结构描述
3. 第4章非业务功能需求：安全性需求 + 稳定性需求的完整文本
4. 第5章系统集成方案：架构设计、网络拓扑、部署设计、技术路线、界面设计的完整文本
5. 第6章接口设计：与外部系统接口 + 内部接口的完整文本
6. 第7章安全性设计：7.1-7.7 所有子节的完整文本

对每个章节，原样保留文本内容，仅标注需要替换项目名称的位置。
```

**Agent K+2（可选）— 技术架构补充**（`subagent_type: Explore`）

仅在有技术架构文档且用户表示架构有变化时启动：

```
读取文件 {arch_path}，提取与过往概设不同的内容：
1. 硬件环境变化（服务器数量/配置调整）
2. 技术栈变化（新增或替换的技术组件）
3. 部署方案变化
4. 安全设计变化
```

### 大文件处理

需求文档和过往概设通常较大。每个 Agent 应使用 Grep 先定位章节位置，再用 Read 的 offset/limit 参数分段读取，避免一次性加载全文。

## Phase 4: 组装输出

概设文档通常超过 2000 行，AI 逐行生成容易丢内容。因此 Phase 4 分两步：先用 AI 写好各子系统的 HLD 分册，再用**脚本确定性拼接**。

### 步骤 4a：生成子系统 HLD 分册

Phase 3 的 Agent 返回了每个子系统的功能提取结果。将每个子系统的内容写入独立的 HLD 分册文件，文件结构与公共章节模板一致：

```
docs/03-系统设计/
├── _common-chapters.md          ← 第1/2/4/5/6/7章公共模板
├── HLD-{子系统名}-v1.0.md      ← 每个子系统一个分册
└── ...
```

每个分册的标题层级使用 `##` 作为章级标题（因为 `#` 留给文档标题）：

```markdown
# {子系统名} 概要设计说明书

## 1 引言
（从公共章节复用，替换 {子系统名称} 占位符为当前子系统名）

## 2 技术选型
（从公共章节复用）

## 3 功能模块设计
### 3.1 功能结构图
### 3.2 功能模块具体功能定义
#### 3.2.1 功能A
#### 3.2.2 功能B
（Phase 3 Agent 提取的功能表格）

## 4-7 章
（从公共章节复用）
```

**占位符替换注意**：公共章节中的 `{子系统名称}` 在分册中替换为对应子系统名。在合并为完整概设时，脚本会用项目简称替换。

### 章节衔接优化

机械拼接的文档容易有"断裂感"。在写公共章节模板（`_common-chapters.md`）时，需要在以下位置加入过渡段落：

| 位置 | 衔接内容 |
|------|---------|
| 第1章末段 | 列出全部子系统名称，承上启下："本文档对XX平台全部N大子系统进行概要设计，涵盖{子系统列表}，为后续详细设计和系统开发提供指导依据。" |
| 第2章末段 | 从技术过渡到功能："基于上述技术架构，以下对各业务子系统的功能模块进行概要设计。" |
| 第3章开头 | 先总述再分述："本章对平台全部N大子系统逐一进行功能模块设计，每个子系统包含模块概述和功能定义表格。" |
| 第3章→第4章 | 从功能过渡到质量："在满足上述功能需求的基础上，系统还需满足以下非业务功能需求。" |

这些过渡段落写在公共章节模板中，脚本拼接时自动带入，不需要 AI 每次临时生成。

读取模板参考文件了解完整的章节骨架和写作规范：
`${CLAUDE_PLUGIN_ROOT}/skills/concept-design/references/template.md`
如果路径变量不可用，使用 Glob 搜索 `**/concept-design/**/template.md` 定位文件。

### 步骤 4b：脚本合并

所有分册写好后，调用插件内置的组装脚本进行**确定性拼接**（不要用 AI 逐行生成 2000+ 行文档，容易丢内容）。

确保 `_common-chapters.md` 的 YAML frontmatter 包含完整的项目元信息（Phase 2 收集的 title/subtitle/company/date/short_name）。

```bash
python3 ${CLAUDE_PLUGIN_ROOT}/scripts/assemble_hld.py docs/03-系统设计/
```

脚本自动完成：
1. 从 `_common-chapters.md` 的 frontmatter 读取项目元信息
2. 自动发现目录下所有 `HLD-*.md` 分册文件
3. 从各分册一级标题提取子系统名
4. 提取各分册第3章功能表格，重编号合并
5. 生成功能结构树（3.1节）
6. 拼接公共章节 + 第3章 → 输出完整概设 `.md`
7. 打印统计信息（总行数、功能点数、预估页数）

`${CLAUDE_PLUGIN_ROOT}/skills/concept-design/references/verification-checklist.md`

- 各章节行数占比

- 如需转换为 Word 交付物，使用 `/md-to-docx` 触发转换

## 写作风格约束

| 维度 | 规范 |
|------|------|
| 人称 | 第一人称复数（"我们"）或无主语被动句 |
| 语气 | 正式、专业、客观 |
| 句式 | 短句为主，避免长复合句 |
| 术语 | 行业标准术语，首次出现缩写需给全称 |
| 编号 | 阿拉伯数字，最多三级（如 3.2.1） |
| 表格 | 首行为表头 |
| 引用 | 书名号《》包裹完整文件名 |

## 质量标准

| 标准 | 要求 |
|------|------|
| 篇幅 | 功能模块设计章节占全文 60% 以上 |
| 独立性 | 不引用"详见PRD第X章"等跳转，所有信息自包含 |
| 一致性 | 模块名称、功能名称与需求确认单完全对齐 |
| 覆盖率 | 需求确认单中每个功能点在概设中都有对应行 |
| 语气 | 第三人称客观描述，不出现技术实现细节 |
| 客户可读性 | 第1章背景、第3章模块简述和功能要求列，非技术人员应能理解。避免代码级术语，用业务语言描述功能 |
