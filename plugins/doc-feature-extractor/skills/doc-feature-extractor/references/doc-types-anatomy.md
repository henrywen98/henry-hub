# 文档类型解剖（Phase 1b 调研产出）

基于 4 类真实样本的结构指纹,后续 extractor 设计参照此文。

## 样本清单

| 类型 | 样本路径 | 用途 |
|---|---|---|
| **PRD** | `PRD-开发/04_PRD_财报解析与分析.docx` | 现有,作回归 |
| **解决方案 (深嵌套)** | `../杭州产投-投资管理系统(二期)建设方案-V11.docx` | 杭州产投建设方案 |
| **解决方案 (浅嵌套)** | `evals/samples/南通-升级方案-V2.4.docx` | 跨项目泛化检验 |
| **报价** | `../杭州资本-二期功能清单报价20250918.xlsx` | xlsx 多 sheet 报价 |

---

## A. PRD .docx 指纹

**章节**: H1/H2/H3 三层,功能模块在 **H2** (例: "流程说明"、"输入"、"输出 - AI回填字段"等)。

**表格** (8 个,关键字段表):
- 表 5: `[字段名称, 字段Key, 类型, 单位, 说明]` — **AI 输出字段表** (12 行)
- 表 7: `[字段名称, 字段Key, 类型, 说明]` — AI 分析字段 (10 行)
- 表 6: `[字段名称, 计算公式, 单位]` — 前端计算字段 (3 行)

**特征载体**: 字段表(`字段名称 / Key / 类型`)是核心抽取对象。也可能有 `FR-XXX` 编号表(其他 PRD 里有,04 这份没有)。

**Extractor 策略**: 现有 `aggregate.py` 的 PRD 模式直接复用 — FR 编号表 + 字段表识别 + 章节切段兜底。

## B. 解决方案 .docx (深嵌套, 杭州产投建设方案)

**章节**: **H2-H6,深 6 层**。
- H2: `项目背景 / 建设目标 / 建设内容 / 技术需求解决方案 / 业务需求解决方案`
- "业务需求解决方案" → H3 (模块组, 例: `国舜市场化基金 / 报告管理 / 履职管理需求 / 流程审批优化`)
- H3 → H4 (功能模块, 例: `主动管理基金 / 项目管理 / 参股基金管理`)
- H4 → H5 (子模块, 例: `新增基金 / 基金详情 / 协议签订`)
- H5 → H6 (字段级, 例: `基本信息 / 已投项目`)

**关键观察**: **"功能模块"在 H4 层级**,不是 H2/H3。如果 fallback heading level 设错(用 H2),会把整份文档当成 5 个功能,完全丢失模块粒度。

**表格**: 几乎没有有效结构化表格(只有版本表、对比表)。**靠章节切段为主**。

**Extractor 策略**: solution-deep 模式 — `fallback_heading_level=4`,关闭 FR 表/字段表识别,纯章节切段。

---

## C. 解决方案 .docx (浅嵌套, 南通升级方案)

**章节**: H1-H4。
- H1: `项目背景 / 核心升级目标 / 升级内容 / 费用测算 / 附录 A/B/C`
- "升级内容" → H2: `AI能力底座搭建 / 核心业务功能完善`
- "核心业务功能完善" → **H3 (~15 个 AI 功能模块, 这就是要抽的!)**:
  `项目BP解析 / 尽调报告辅助生成 / 法务审查报告辅助生成 / 合同智能审查 / 合同智能解析 / 财报识别 / 财报智能分析 / 报告智能生成 / 月度业务报表自动生成 / AI助手 / 系统架构升级 / 系统功能升级 / 项目关键时间节点提醒 / 投后估值系统 / 风险预警 / 自定义报表 / 产控集团系统对接`
- H3 → H4 (子模块, 例: `项目筛选 / 资金流数据自动同步`)

**表格** (9 个,**有黄金表 3**!):
- **表 3**: `[序号, 功能模块, 功能模块, 预估费用(万元), 备注, 预估工作量(人月)]` — 19 行 **内嵌报价表**
- **表 5**: `[AI 智能体, 输入, 核心能力, 输出]` — 5 行 **能力清单**
- 表 6: `[用途, 推荐模型, 参数量, FP8 显存, 理由]` — 模型选型
- 表 1/7/8: 硬件方案 / 配置 / 预算 — 硬件成本

**关键观察**:
1. 功能模块在 **H3 层级** (不是 H4 — 跟杭州建设方案不同),所以 solution 模式必须**让用户配 fallback_heading_level**
2. 内嵌"功能模块 + 费用 + 工作量"表 — 这是个金矿,可以直接当 quotation 表抽
3. 内嵌"AI 智能体"表 — 可以当 capability 来源

**Extractor 策略**: solution-shallow 模式 — `fallback_heading_level=3`。如果识别到表头含 `功能模块 + (费用|工作量|工时)`,优先抽该表为模块清单。

## D. 报价 .xlsx (杭州资本)

**结构**: 9 个 sheet,跨多个客户 (产投 / 科创 / 人才):
- 总览类: `汇总报价` (12 行,公司 × 分类 × 报价)
- 详细报价类: `报价1 / 杭州科创-报价清单 / 功能报价2-备份 / 杭州产投-报价清单 / 杭州人才-报价清单`
- 功能清单类(无价): `杭州科创-招标功能 / 杭州产投-招标功能 / 杭州人才-招标功能`

**报价 sheet 列结构** (稳定): `[序号, 业务范围, 模块, 子模块, 功能描述, 分摊报价]` (+ 可选 `优先级 / 备注`)

**功能清单 sheet 列结构**: `[序号, 业务范围, 功能模块, 二级功能, 功能描述]` (无价)

**末尾**: 通常有"合计"行作为终止标记。

**关键观察**:
1. 行式数据,每行 = 一个功能子模块,直接对应 `module_overview` sheet 的一行
2. **业务范围 / 模块 / 子模块** 三级结构,跟 PRD 抽出来的 `(前缀, 子模块)` 概念完全对得上
3. **分摊报价** 列是金钱信息,新 quotation extractor 必须保留这一列到输出
4. 可能有"小计"/"合计"行需要过滤掉

**Extractor 策略**: quotation 模式 — openpyxl 读 sheet,识别表头(`功能描述 + 分摊报价` 是双列指纹),逐行抽,跳过合计行。每个 sheet 当一个独立"前缀"或"业务范围"处理。

## 跨样本设计要点

| 维度 | PRD | Solution-deep | Solution-shallow | Quotation |
|---|---|---|---|---|
| 输入格式 | .docx | .docx | .docx | .xlsx |
| 模块在哪 | H2 + 字段表 | H4 章节 | H3 章节 + 内嵌报价表 | 行式数据 |
| 报价信息 | 无 | 无 | 表 3 内嵌 | 主要载体 |
| 字段定义 | 字段表 | 无 | 无 | 无 |
| 是否 FR 编号 | 通常有 | 无 | 无 | 序号(非语义) |
| 适配 sheets | 全 6 sheet | 模块概览 + 章节明细 | 模块概览 + 报价对照 | 模块概览 + 报价对照 |

**结论**: 
- 抽取引擎必须把 `fallback_heading_level` 提升为**每文档(或每前缀)可配**,不能只按前缀
- 新增"内嵌报价表识别"逻辑(表头含 `功能模块 + 费用|工作量|工时|报价`)
- xlsx 输入要全新 extractor,不能套现有 docx 流程
- "AI 字段定义" sheet 仅 PRD 启用;"报价对照" sheet 仅 quotation/solution-shallow 启用
