# 需求确认单编写规则

## 零、颗粒度定位（最高优先级）

本文档定位为"需求确认单"，目的是与客户对齐需求范围和关键业务规则，不是PRD/详细设计。

文档受众：客户方的分管领导和业务对接人。他们关心的是"要做哪些功能、范围是什么、关键业务规则是否正确"，不关心字段类型、UI组件、交互细节等实现层面的内容。输出的每一句话都要经得起这个检验——"这是领导签字时需要确认的信息吗？"

颗粒度原则——无论用户输入多详细或多粗略，输出都必须锁定在以下层级：

✅ 需求确认单该写的：
- 功能做什么（业务目标和范围）
- 涉及哪些信息项（必须用两列表格列出：信息项 + 备注，这是每个模块的必输出内容）
- 核心业务规则和约束（审批流向、关键联动逻辑、特殊取值规则）
- 与现有功能的变更差异（新增/修改/删除了什么）

❌ 需求确认单不该写的（留给PRD阶段）：
- 字段标识、接口名等技术标识（如 name、shortName、scope）
- 字段类型、是否必填、交互说明、显隐条件等实现细节
- 具体的UI组件描述（下拉单选、文本域、日期选择器等）
- 页面级的CRUD拆分（不需要分列表/新增/编辑/详情四个小节）
- 数据校验规则、默认值、模糊搜索等实现细节

颗粒度校验：写完每个模块后自问——"这段内容是在确认'做不做、做什么'，还是在定义'怎么做'？" 如果是后者，删掉或概括。

## 一、输出结构（严格遵循）

每个功能模块按以下结构输出，模块之间用分隔线隔开：

```markdown
## X. {模块名称}

### X.1 功能概述

入口：{菜单路径}

{两段式：是什么 + 为什么}

### X.2 功能要点

用编号列表描述该模块的核心功能点，每条一句话说清楚"做什么"。

### X.3 信息项清单

| 信息项 | 备注 |
|-------|------|
| {名称} | {需要客户了解的说明} |
| 其他信息项 | {无特殊备注的信息项名称，用顿号分隔} |

### X.N 审批流程
{仅在涉及审批时输出此节}

### X.N 特殊业务规则
{仅在涉及自动计算、条件取值、状态流转等复杂逻辑时输出此节}
```

说明：X.1功能概述、X.2功能要点、X.3信息项清单为固定小节，后续小节（审批流程、特殊业务规则）按实际存在的内容顺序编号。

## 二、编写规则

### 功能概述规则

- 入口在前，正文在后：先写"入口：{菜单路径}"，再写概述正文
- 正文控制在100-200字，两段式结构：
  (1) 是什么：一句话概括此功能的核心能力
  (2) 为什么：说明业务价值或解决的痛点
- 禁止使用技术术语（如"API调用""组件渲染""接口"）
- 用客户能理解的业务语言

### 功能要点规则

- 每个要点一句话，聚焦"做什么"而非"怎么做"
- 功能要点只写行为和规则变化（如"加了XX功能""XX情况下需要上传材料"），不要重复描述信息项的改名、来源等——这些由信息项清单表格承载
- 条目数量控制在3-6条，避免事无巨细

### 信息项清单规则

- 固定两列：信息项 | 备注
- "信息项"列写信息的名称，不写字段类型（不写"下拉""文本框"等）
- "备注"列只写客户需要了解的业务含义，比如：
  · 新增的信息项 → 写"新增"
  · 修改过的信息项 → 写"原叫XX，现改为XX"
  · 有特殊来源的 → 写"系统自动带出"
  · 有显示条件的 → 写"只有XX情况才显示"
- 表格分两部分展示：
  · 表格主体：只列有特殊备注的信息项（新增、改名、系统自动带出、有条件显示等），每项单独一行
  · 表格末尾一行：合并所有无特殊备注的信息项，写成"其他信息项 | XX、XX、XX等"
- 如果多个信息项的备注完全相同（如7个都是"系统自动带出，不能改"），合并为一行说明，不要逐条重复
- 不要在备注里写UI组件类型、是否必填、校验规则等实现细节
- 表格只列主要信息项，不需要穷举每一个字段（如果信息项超过15个，用"等"收尾）

### 审批流程规则

- 仅在功能涉及审批时才输出此节
- 用箭头表示流转：{角色A} → {角色B}（{动作概述}） → {角色C}（{动作概述}）
- 如果存在多条流程路径，先写明分支条件，再分别列出各路径：
  · 分支条件举例：按业务类型（项目/基金）、按金额区间、按审批级别等
  · 用①②编号区分不同路径，每条路径前注明触发条件
- 只有一条流程时，直接写箭头链，不需要编号
- 括号内只写该节点的关键动作概述，不需要逐字段展开

### 特殊业务规则

- 仅在涉及自动计算、条件取值、状态流转等需要客户确认的复杂逻辑时输出
- 用自然语言概述规则，避免写成伪代码或公式
  · 正确示例："项目状态根据审批进度自动流转，区分权限内项目和权限外项目两套状态路径"
  · 错误示例：逐条罗列每个状态值的枚举
- 如果规则较复杂，可标注"详见PRD"

### 计划外需求标注

- 超出合同范围的需求，在模块标题后加"-⚠️计划外"
- 示例：## 3. 字段优化-⚠️计划外

### 语言规范

- 全文使用简体中文输出
- 用大白话写，让不懂技术的客户领导一遍就能看懂：
  · 写"系统自动带出" 而非 "接口回填"
  · 写"上传附件" 而非 "触发file input"
  · 写"展示/隐藏" 而非 "组件显隐控制"
  · 写"填写" 而非 "录入表单字段"
  · 写"选了A才出现B" 而非 "A字段值触发B字段显隐联动"
  · 写"系统自动算出来的，不能手动改" 而非 "系统自动生成，不支持修改"
- 一句话只说一件事，长句拆短句。如果一条功能要点超过两行，就拆成两条
- 避免堆砌名词短语，能用动宾结构就用动宾结构：
  · 写"选完企业后，系统会自动带出对应的平台信息" 而非 "企业选择后平台信息自动带出"
- 术语全文统一，不混用（如统一用"附件"而非有时写"文件"、有时写"附件"）

## 三、输入处理策略

根据用户输入类型，采取不同策略：

1. 会议纪要/调研笔记 → 提取需求点，按功能模块分组，概括为功能要点 + 提取信息项填入表格
2. 需求清单 → 按条目归类到功能模块，展开为功能要点 + 提取信息项填入表格
3. 原型截图描述 → 从界面元素概括功能范围，提取信息项填入表格
4. 混合内容 → 综合以上策略
5. 非常详细的输入（含字段表格、字段标识、交互细节等） → 信息项名称保留并填入两列表格，但去掉技术标识（如字段ID）、字段类型、交互细节等实现层面的信息

关键原则：无论输入多详细，输出格式和颗粒度不变。输入中的信息项要转化为两列表格（信息项+备注），不是丢掉不写，而是去掉技术细节只保留业务含义。

当信息不完整时：
- 菜单入口不明确 → 功能概述的入口写"【待确认：具体菜单路径】"
- 审批流程角色不确定 → 列出流程结构，角色名标注"【待确认】"
- 关键业务规则不确定 → 标注"【待确认】"
- 绝不编造具体业务数据或码值

## 四、参考示例

以下示例中的模块编号仅为演示，实际使用时按文档中的模块顺序编排。

### 示例A：标准功能模块

```markdown
## 1. 委派人员管理

### 1.1 功能概述

入口：履职管理-委派人员管理

管理公司向基金、项目委派的董事、监事、高管等人员信息。支持新增委派和人员变更，配合审批流程确保合规。

### 1.2 功能要点

1. 列表页展示委派人员信息，支持新增、编辑、删除。
2. （新增）加一个"类型"来区分是"新增委派"还是"变更委派"。选了变更的话，需要关联到变更前的人员。
3. 基金及专项子基金会显示"公司章程/基金合伙协议"，直投项目不显示。如果涉及工商变更，需要上传变更材料。

### 1.3 信息项清单

| 信息项 | 备注 |
|-------|------|
| 委派人员 | 原叫"姓名"，现改为"委派人员" |
| 委派职务 | 原叫"岗位类型"，现改为"委派职务" |
| 类型（新增） | 区分"新增委派"和"变更委派" |
| 变更前人员（新增） | 选了"变更"才显示 |
| 所属平台（新增） | 系统自动带出 |
| 工商变更材料 | 选了"是"才需要上传 |
| 公司章程/基金合伙协议 | 只有基金及专项子基金显示 |
| 其他信息项 | 代表公司、被投企业、年度、届数、委派开始时间、委派结束时间、是否需要申报、是否工商变更、相关附件 |
```

### 示例B：含审批流程的功能模块

```markdown
## 4. 市国资委备案

### 4.1 功能概述

入口：项目/基金管理-市国资委备案

管理基金和项目的国资委审批备案，记录备案类型、出资企业、批复时间等信息，确保投资决策合规。

### 4.2 功能要点

1. 列表页支持新增、编辑、删除。（新增）加一列"状态"。
2. 国资委批复时间和备案相关文件不是申请人填的，而是在审批环节由经办人录入。

### 4.3 信息项清单

| 信息项 | 备注 |
|-------|------|
| 国资委批复时间 | 审批环节由经办人录入 |
| 相关文件 | 审批环节由经办人上传 |
| 状态（新增） | 新增 |
| 其他信息项 | 备案类型、基金创投经办人、企业名称、出资企业名称、审核内容 |

### 4.4 审批流程

按业务类型区分两条路径：
①项目：投资经理 → 投资发展部经办人（录入批复时间、上传备案文件）
②基金：投资经理 → 基金管理部经办人（录入批复时间、上传备案文件）
```

### 示例C：含特殊业务规则的功能模块

```markdown
## 1. 重点项目推进表

### 1.1 功能概述

入口：首页-通知中心-我的任务

每周跟踪直投项目和基金的推进情况。系统自动汇总本周的进展数据，投资经理填写主观进展和后续计划，方便管理层及时了解重点项目动态。

### 1.2 功能要点

1. 任务标题改为"重点项目(直投项目)每周推进情况"/"重点项目(基金)每周推进情况"。线上的历史任务数据删除。
2. 处理页面分两部分：一部分是系统自动带出的项目基本信息，另一部分是投资经理手动填写的进展信息。
3. "本周进展"分两块：系统自动生成的和投资经理手动填的。手动填写部分会默认带出上周"下周及后续计划"的内容。
4. 页面数据在周日24:00之前是动态更新的，之后就固定下来不再变化。

### 1.3 信息项清单

以下信息项由系统自动带出，不能手动修改：项目名称/简称、我方拟出资渠道、我方拟投资金额、投前估值（万元）、所属产业链、项目来源、A/B类。

| 信息项 | 备注 |
|-------|------|
| 当前存在的问题和需要协调的事项 | 投资经理填写，默认带出上周内容 |
| 本周进展-自动生成 | 系统自动生成，不能改 |
| 本周进展-投资经理填写 | 默认带出上周"下周及后续计划"的内容 |
| 下周及后续计划 | 投资经理填写 |

### 1.4 特殊业务规则

- 项目状态会根据审批进度自动变化，分两种情况：权限内的项目走一套状态（从"未立项"到"已打款"共6步），权限外的项目因为要多过投审会、党委会、董事会等环节，走另一套状态（共9步）。具体每一步的状态名称见PRD。
```
