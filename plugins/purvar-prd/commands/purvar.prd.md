## 用户输入

```text
$ARGUMENTS
```

必须在开始之前检查用户输入。如果为空或 `$ARGUMENTS` 显示为字面值，提示用户在 `/purvar.prd` 后面粘贴需求素材。

## 任务

作为资深产品经理，将用户提供的原始需求素材（会议纪要、需求清单、原型页面描述、需求确认单或混合内容），转化为开发和测试可以直接使用的功能迭代型PRD。

文档受众：开发工程师和测试工程师。开发需要据此拆任务写代码，测试需要据此写用例做验收。输出的每一条内容都要经得起这个检验——"开发看了知道怎么做吗？测试看了知道怎么验吗？"

## 执行步骤

### 1. 加载编写规则

读取 `${CLAUDE_PLUGIN_ROOT}/references/prd-rules.md`，严格遵循其中的：
- **颗粒度定位**：PRD该写什么、不该写什么
- **输出结构**：章节顺序和格式
- **编写规则**：每个章节的具体规范
- **输入处理策略**：根据输入类型采取对应策略

### 2. 解析输入

识别输入类型：
- 会议纪要/调研笔记
- 需求清单
- 原型截图描述
- 需求确认单
- 混合内容

按 prd-rules.md 中"输入处理策略"采取对应策略。

### 3. 生成PRD内容

按 prd-rules.md 中的"输出结构"生成完整PRD：

1. **原型表格**：列出涉及的页面，链接如有则填入，否则写"【待补充】"
2. **变更概要**：业务诉求 + 方案概述，各一句话，合计不超100字
3. **影响范围**：列出所有涉及的模块/页面，标注变更类型（新增/修改/联动影响）
4. **功能说明**：按页面拆分子章节，每个包含入口、业务规则、操作说明、六列信息项表格
5. **异常与边界**：覆盖数据冲突、权限边界、输入边界、状态边界
6. **验收标准**：checkbox格式，具体可验证，覆盖正向和异常流程
7. **开放问题**：记录所有【待确认】项（如有）

### 4. 信息处理原则

- 输入中有明确信息 → 如实填写
- 能合理推断的（如"金额"的类型是数字输入）→ 按常识推断，备注标注"推断"
- 无法推断的 → 写"【待确认】"，同时记录到开放问题
- **绝不编造**具体的业务数据、枚举值或校验规则

### 5. 确定文件名并保存

从需求素材中提取核心功能名称（中文，2-8字），将PRD写入当前工作目录：

```
PRD-{功能名称}.md
```

如果同名文件已存在，在名称后加序号（如 `PRD-委派人员管理-2.md`）。

### 6. 报告完成

输出：
- 生成的文件路径
- PRD包含的功能模块数量
- 【待确认】标记数量（如有）
- 如有待确认项，提示可使用 `/purvar.clarify` 进一步完善
