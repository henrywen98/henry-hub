description: 识别PRD中的模糊区域，通过逐题交互式提问澄清并更新文档
---

## 用户输入

```text
$ARGUMENTS
```

## 概述

目标：检测并减少当前PRD中的歧义或缺失项，通过交互式提问澄清后直接更新PRD文件。

## 执行步骤

### 1. 定位PRD文件

按以下优先级确定要澄清的PRD文件：

1. 如果用户在 `$ARGUMENTS` 中提供了文件路径 → 使用该路径
2. 如果用户输入为空 → 在当前目录搜索 `PRD-*.md` 文件
   - 找到1个 → 直接使用
   - 找到多个 → 列出让用户选择
   - 找不到 → 提示用户先运行 `/purvar.prd` 生成PRD

### 2. 加载扫描分类

读取 `${CLAUDE_PLUGIN_ROOT}/references/clarify-categories.md`，获取7个扫描维度和具体检查项：

1. 变更概要
2. 影响范围
3. 信息项完整性
4. 业务规则
5. 异常与边界
6. 验收标准
7. 开放问题

### 3. 结构化扫描

加载PRD文件，对每个维度进行结构化扫描。标记状态：

| 状态 | 含义 |
|------|------|
| ✅ 清晰 | 信息完整，无需澄清 |
| ⚠️ 部分 | 有可改进的地方 |
| ❌ 缺失 | 关键信息缺失，需要澄清 |

生成内部覆盖图用于优先级排序（不直接输出原始图）。

对每个状态为"部分"或"缺失"的维度，生成候选澄清问题，除非：
- 澄清不会实质性改变开发实现或测试验证
- 信息更适合留到技术方案阶段

### 4. 生成优先级问题队列

内部生成最多5个优先级问题。约束：

- 整个会话最多 5 个问题
- 每个问题必须可通过以下方式回答：
  - 简短的多选选择（2-5 个互斥选项），或
  - 一词/短语答案（5 个词以内）
- 仅包含答案会实质性影响开发拆任务、测试写用例、业务规则准确性的问题
- 确保维度覆盖平衡：优先覆盖 ❌ 缺失 的维度
- 排除琐碎的风格偏好或技术实现细节

### 5. 逐题提问循环

每次仅展示**一个**问题。

**多选问题格式**：

1. 分析所有选项，确定**最合适的选项**
2. 在顶部展示推荐：`**推荐：** 选项 [X] - <理由>`
3. 渲染选项表格：

| 选项 | 描述 |
|------|------|
| A | <选项 A 描述> |
| B | <选项 B 描述> |
| C | <选项 C 描述> |
| 简答 | 提供不同的简短答案（5 个词以内） |

4. 表格后添加：`回复选项字母（如 "A"），说"是"或"推荐"接受推荐，或提供自己的答案。`

**简答问题格式**：

1. 提供建议答案：`**建议：** <答案> - <理由>`
2. 添加：`格式：简短答案（5 个词以内）。说"是"或"建议"接受建议，或提供自己的答案。`

**用户回答后**：
- "是"/"推荐"/"建议" → 采用推荐/建议答案
- 选项字母 → 采用对应选项
- 其他 → 验证是否符合约束，有歧义则追问

**停止条件**：
- 所有关键歧义提前解决
- 用户表示完成（"完成"、"好了"、"没有了"）
- 达到 5 个已问问题

### 6. 即时集成（每个答案后）

接受答案后立即更新PRD文件：

**澄清记录**：
- 确保PRD中存在 `## 澄清记录` 章节（如缺失，在变更概要之后创建）
- 在其下创建（如不存在）`### 会话 YYYY-MM-DD` 子标题
- 追加要点：`- 问：<问题> → 答：<最终答案>`

**正文更新**：
- 变更概要类澄清 → 更新变更概要章节
- 信息项类澄清 → 更新对应信息项表格（填入类型/校验/默认值，替换"【待确认】"）
- 业务规则类澄清 → 更新或补充功能说明中的业务规则
- 异常场景类澄清 → 补充异常与边界表格
- 验收标准类澄清 → 补充或修改验收标准
- 开放问题类澄清 → 更新开放问题状态为"已确认"并填写结论

**更新原则**：
- 如果澄清使早期模糊陈述无效，替换该陈述而非重复
- 不留过时的矛盾文本
- 保持格式不变，不重新排序无关章节
- 每次集成后保存文件

### 7. 验证（每次写入后）

- 澄清记录中每个接受答案恰好一个要点（无重复）
- 更新的章节不包含已回答问题对应的"【待确认】"标记
- 无矛盾的早期陈述残留
- Markdown结构有效

### 8. 报告完成

提问循环结束后输出：

- 已问和已答问题数
- 更新后PRD的路径
- 涉及的章节列表
- 覆盖摘要表：

| 维度 | 状态 |
|------|------|
| 变更概要 | ✅ 清晰 / ⚠️ 已改善 / ❌ 待处理 |
| 影响范围 | ... |
| 信息项完整性 | ... |
| 业务规则 | ... |
| 异常与边界 | ... |
| 验收标准 | ... |
| 开放问题 | ... |

- 如有待处理或延迟项，建议是否需要再次运行 `/purvar.clarify`

## 行为规则

- 如果未发现有意义的歧义，回复："未检测到需要澄清的关键问题。"并建议继续后续工作。
- 永远不超过 5 个已问问题。
- 避免技术栈层面的问题（如"用什么框架""接口怎么设计"）。
- 尊重用户提前终止信号（"停止"、"完成"、"好了"）。
- 不提前透露后续排队的问题。
