# PM 碎片信息收集与洞察整理

将来自多渠道的碎片化信息整理为结构化的需求洞察文档，作为后续 PRD / 用户故事 / Issue 拆解的输入源。

## 定位与边界

这个 skill 处于 PM 工作流的**上游**——在需求正式成型之前：

```
碎片信息（本 skill）→ 结构化洞察 → PRD / 用户故事 / Issue 拆解
     ↑                    ↑               ↑
  多渠道收集           本 skill 输出     下游工具处理
```

**是什么**：碎片 → 结构化洞察的整理器
**不是什么**：不生成 Issue（用 `/req-to-issues`），不写 PRD（用 `purvar-prd` 或 `create-prd`）

## 为什么需要分离问题和方案

用户反馈中经常混杂"问题"和"方案建议"。比如用户说"能不能加个导出按钮"，真正的需求可能是"我需要把数据分享给同事"。如果直接把"加导出按钮"作为需求，就限制了解决方案的空间。本 skill 的核心价值之一就是从用户提出的方案中提炼出背后的真实问题（Opportunity），为后续决策保留更大的设计空间。

## Step 1: 信息收集与预处理

### 接收原始内容

接受任意格式的碎片信息，常见来源：

| 来源类型 | 典型形式 |
|---------|---------|
| 客户会议 | 会议纪要、语音转文字 |
| 即时通讯 | 微信群聊天记录、飞书消息 |
| 客服渠道 | 工单、投诉邮件、电话记录 |
| 用户访谈 | 访谈记录、问卷回复 |
| 竞品分析 | 竞品功能对比笔记 |
| 内部讨论 | 团队讨论记录、Slack 消息 |
| 混合来源 | 用户直接粘贴的多条混合内容 |

### 预处理

1. 识别信息来源类型，标注在输出中
2. 如果内容过短（<50字）或缺乏上下文，追问补充：
   - "这些反馈来自什么场景？（客户会议/线上反馈/内部讨论）"
   - "涉及哪个产品/模块？"
   - "反馈的用户是什么角色？"

---

## Step 2: 主题聚类与场景提取

### 按业务主题聚类

不使用简单的 Bug/优化/新需求分类，而是按**业务主题或模块**聚类。比如"订单提交慢"和"订单无法导出"归入同一个"订单管理"主题，而不是分别归为"Bug"和"新需求"。

### 提取用户场景（JTBD）

对每条反馈，提取背后的用户场景，用 Jobs-to-be-Done 格式表达：

> **When** [具体情境]，**I want to** [动机/目标]，**so I can** [期望结果]

例：用户说"审批流程太复杂"→ JTBD: When 我需要提交一个紧急采购申请时，I want to 快速完成审批流程，so I can 不耽误采购进度。

### 分离问题与方案

对每条反馈判断：
- **问题（Opportunity）**：用户遇到的困难、痛点、未满足的需求
- **方案建议（Solution）**：用户提出的具体解决方案

如果用户直接给的是方案建议，追溯到背后的问题。在输出中两者都保留，但以问题为主线组织。

---

## Step 3: 洞察提炼

对每个主题聚类，从以下维度提炼洞察：

### 核心问题描述

用 Opportunity 语言（描述问题/机会），不用 Feature 语言（描述解决方案）：
- ✅ "用户在批量处理订单时效率低下"
- ❌ "需要增加批量操作按钮"

### 情感标注

标注反馈的情感倾向，帮助判断紧迫程度：

| 标注 | 含义 | 信号词 |
|------|------|-------|
| 🔴 负面 | 用户有明显不满或受阻 | "无法"、"太慢"、"经常出错"、"很烦" |
| 🟡 中性 | 用户提出改进建议但无强烈情绪 | "能不能"、"最好"、"建议" |
| 🟢 正面 | 用户认可现有功能或方向 | "挺好的"、"继续保持"、"比之前好多了" |

### 影响面评估

| 维度 | 评估方式 |
|------|---------|
| 用户量 | 影响个别用户 / 某类用户 / 全体用户 |
| 频次 | 偶发 / 日常操作 / 核心路径 |
| 严重程度 | 不便 / 效率降低 / 功能不可用 |

### Opportunity Score

参考 Dan Olsen 的机会评分模型：

```
Opportunity Score = 重要性 × (1 - 满意度)
```

- **重要性**（0-1）：这个需求对用户完成核心任务有多关键？
- **满意度**（0-1）：用户对当前方案的满足程度？

得分越高，代表这个机会越值得投入。无法精确量化时，用 高/中/低 定性估计。

---

## Step 4: 优先级排序与输出

### 排序逻辑

1. 首先按 Opportunity Score 降序排列
2. 同分时，负面情感的优先于中性
3. 影响全体用户的优先于影响个别用户

### 生成洞察文档

读取输出模板了解完整的文档结构：
`${CLAUDE_PLUGIN_ROOT}/skills/ai-pm-feedback-collector/references/output-template.md`

文档包含：
1. **信息来源摘要**：来源渠道、时间、涉及产品/模块
2. **主题聚类**：每个主题下的核心问题、JTBD、原始反馈引用、情感标注、影响面、Opportunity Score
3. **优先级排序表**：全局排序一览
4. **待验证假设清单**：每个高优先级项的关键假设及建议验证方式

### 输出文件

- 文件名：`YYYY-MM-DD-反馈洞察-{主题关键词}.md`
- 写入项目根目录（或用户指定路径）
- 如果用户在同一项目中多次运行，询问是否追加到已有文件

---

## Step 5: 下游衔接建议

整理完成后，根据洞察结果向用户建议下一步行动：

| 洞察结果 | 建议动作 | 衔接工具 |
|---------|---------|---------|
| 高优先级 Bug / 明确需求 | 直接拆解为 Issue | `/req-to-issues` |
| 需要深入了解的 Opportunity | 设计用户访谈 | pm-skills `interview-script` |
| 需要量化验证的假设 | 设计实验方案 | pm-skills `brainstorm-experiments` |
| 多个洞察指向同一方向 | 整合为产品需求文档 | `purvar-prd` 或 pm-skills `create-prd` |
| 需要了解竞品做法 | 竞品分析 | pm-skills `competitor-analysis` |
| 需要确定优先级策略 | 特性优先级排序 | pm-skills `prioritize-features` |

以对话方式呈现："根据整理结果，我建议接下来可以…… 你想先做哪个？"
