# 规范性保障

## 一、项目进度控制

### 1.1 进度控制原则

为保证管理应用系统的成功实施，必须按照系统开发流程对项目实施进行严格项目进度控制。项目启动前，各方人员应做好充分准备，确保项目所需人员及其他必要条件及时就绪。项目实施过程由用户相关单位和部门与我公司共同完成，各方应通力合作，并保证人员稳定。

公司有专门的项目进度控制部门，并且有着严格的项目管理流程，对项目进度和质量进行实时的监督与管理。

### 1.2 进度控制方法

1. 项目经理必须根据任务情况做好进度安排计划，按周做计划以书面呈交项目协调委员会
2. 以周为单位做计划以书面形式下达各组，各组分头安排落实到个人
3. 组长或个人在接到计划书时，认为恰当则签字；若认为不恰当，必须及时陈述理由
4. 在计划时间到时，项目经理严格按照进度计划书验收
5. 若验收不合格，则责成3日内修正，若仍不能完成必须以书面形式说明理由
6. 质量保证组作为项目组外的成员，客观地监控和评价项目的进展情况，及时发现和指出项目中存在的或潜在的问题

## 二、项目范围控制

### 2.1 范围管理内容

项目范围管理通过规划范围管理、收集需求、定义范围、创建WBS、确认范围和控制范围六个过程来实现。

- **确认范围**：正式验收项目已完成的可交付成果的过程，由外部干系人（客户或发起人）对项目可交付成果进行检查验收
- **控制范围**：监督项目和产品的范围状态、管理范围基准变更过程，在整个项目期间保持对范围基准维护

### 2.2 范围管理与质量控制区分

- 确认范围主要强调可交付成果获得客户或发起人的接受；质量控制强调可交付成果的正确性
- 质量控制一般在确认范围前进行，也可同时进行
- 质量控制属内部检查；确认范围是外部干系人对可交付成果进行检查验收

## 三、需求变更管理

### 3.1 需求分级管理

对于项目需求，实行分级管理，以达到对需求变更的控制和管理：

| 级别 | 名称 | 说明 | 处理方式 |
|------|------|------|---------|
| 一级 | Urgent | 关键性需求，不满足则整个项目不能正常交付 | 必须立即满足 |
| 二级 | Necessary | 后续关键性需求，不满足则新内容无法继续 | 必须满足，可排优先级 |
| 三级 | Needed | 后续重要需求，不满足则项目价值下降 | 应该实施，可排优先级 |
| 四级 | Better | 改良性需求，不影响已有功能使用 | 时间资源允许时实施 |
| 五级 | Maybe | 可选性需求，属个人喜好或设想 | 时间资源允许时考虑 |

### 3.2 全生命周期变更管理

项目生命周期分为三个阶段，需求变更管理贯穿全过程：

**1）项目启动阶段的变更预防**
- 从需求分析入手，力争基准文件定义的范围详细清晰
- 对不明确的需求尽可能与客户沟通，挖掘真实需求
- 需求分析完成后与客户一同进行评审，评审通过后签字确认

**2）项目实施阶段的需求变更**
- 分析变更请求，评估变更可能带来的风险和修改基准文件
- 需求的变更要经过出资者的认可
- 小的需求变更也要经过正规的需求管理流程
- 项目经理作为中间关键协调人，注意沟通技巧

**3）项目收尾阶段的总结**
- 项目总结工作包括风险应对措施的分析和总结
- 项目中发生的变更和问题的分析统计的总结

### 3.3 需求变更管理原则

1. **建立需求基线**：需求基线是需求变更的依据，每次变更并经过评审后都要重新确定新的需求基线
2. **制订变更控制流程**：简单、有效的变更控制流程，并形成文档
3. **成立CCB**：项目变更控制委员会由项目多方人员共同组成，包括用户方和开发方的决策人员
4. **先申请再评估**：需求变更要先申请然后再评估，经过与变更大小相当级别的评审确认
5. **同步更新**：需求变更后，受影响的软件计划、产品、活动都要进行相应的变更
6. **妥善保存**：变更产生的相关文档妥善保存

### 3.4 需求变更应对方法

通过现场需求调研、联络会等方式明确项目需求变更内容，并以合同或会议纪要形式加以明确；当需求与标准化设计成果不相符合时，需将变更需求提总部审批通过后再开展实施。

## 四、项目配置管理

### 4.1 配置管理总则

按照CMMI5中配置管理的要求，对所有的工作产品进行控制和管理，保证在项目整个生命周期中工作产品的完整性和可追溯性。控制和管理的内容包括配置项的标识、基线的建立、变更控制和版本控制等。此项工作由专门的配置管理组承担。

### 4.2 项目投标阶段配置管理

以调研考察用户需求为核心，通过与用户的接触和充分沟通，适时形成并评审《用户需求规格说明书》《项目解决方案》《合同草案》等文档。经客户和开发双方评审认可的《需求规格说明书》将作为项目的基线进行管理。

变更控制过程：提出书面变更请求 -> SCCB审核 -> 分派实施变更任务 -> 完成情况评审 -> 纳入需求基线管理。

### 4.3 项目实施阶段配置管理

以投标阶段的文档为基础，重新配置实施过程中需要的资源。通过WBS进行任务的明确和职责划分，通过《项目开发策划》《项目测试计划》《项目验收方案》进行阶段控制。

### 4.4 文档分类管理

所有文档可划分为两种类型：
- **基础文档（必须有的文档）**：每一份文档都是项目行进中的一个里程碑，标志了前一阶段的结束和下一阶段的开始
- **辅助文档（根据情况写的文档）**：项目进行中的记录性文档，作为项目进度的控制和阶段考评的重要依据

## 五、代码管理规范

### 5.1 高效的版本控制

使用主流版本控制系统Git工具，实现多团队并行开发无冲突、代码变更可追溯、版本迭代可管控，大幅提升开发协同效率，降低系统上线风险。

### 5.2 标准化分支管理策略

采用"主分支+功能分支+发布分支+热修复分支"的分层分支模型：

| 分支类型 | 命名规范 | 用途 |
|---------|---------|------|
| 主分支（main/master） | main | 保持随时可部署的稳定状态，仅接收发布分支或热修复分支的合并，禁止直接提交代码 |
| 开发分支（develop） | develop | 日常开发主分支，整合各功能分支代码，持续迭代更新 |
| 功能分支 | feature/业务模块-功能名称 | 基于develop创建，单功能开发完成后通过PR提交审核 |
| 发布分支 | release/版本号 | 基于develop创建，仅修复测试问题，不新增功能 |
| 热修复分支 | hotfix/问题描述-版本号 | 基于main创建，用于紧急修复生产环境缺陷 |

### 5.3 规范化提交与协作流程

- **提交信息标准化**：采用"类型(模块): 描述"的提交格式（如feat(债权管理): 新增保理业务审批流程），类型包括feat、fix、docs、refactor等
- **PR审核机制**：功能分支合并前必须提交PR，指定至少2名核心开发人员进行代码审查，审核通过后方可合并
- **冲突提前规避**：开发过程中要求每日同步develop分支最新代码，通过CI/CD工具自动检测冲突风险

### 5.4 全链路追溯与安全管控

- **变更全程追溯**：每一次代码提交、分支合并、PR审核均记录操作人、时间、变更内容，支持通过版本号快速回滚
- **权限精细管控**：基于角色分配仓库访问权限，开发人员仅能操作自身负责的分支，管理员掌握主分支保护、合并权限
- **集成自动化工具**：对接CI/CD流水线，代码提交后自动触发单元测试、代码质量扫描（如SonarQube），未通过则禁止合并

### 5.5 清晰的代码注释

**注释核心原则**：
- 精准实用：聚焦代码"为什么做""怎么做"，不重复代码逻辑
- 同步更新：代码修改时必须同步更新对应注释
- 简洁规范：语言简洁明了，格式统一
- 覆盖关键：优先注释核心业务逻辑、复杂算法、特殊处理

**分层注释规范**：
- 文件级注释：每个代码文件头部必须包含文件级注释
- 类/接口级注释：说明职责、设计初衷及核心作用
- 方法级注释：详细说明功能、参数含义、返回值、异常情况
- 代码块级注释：针对复杂逻辑、关键算法添加行内或块级注释

**注释落地保障**：
- 工具辅助：集成SonarQube、CheckStyle，配置注释覆盖率阈值（核心业务代码注释覆盖率>=80%）
- 模板统一：提供文件、类、方法级注释模板，嵌入IDE
- 代码评审把关：将注释质量纳入代码评审标准
