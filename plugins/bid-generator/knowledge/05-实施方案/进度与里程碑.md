# 进度与里程碑

## 一、通用阶段划分模板

整个项目开发周期划分为以下阶段，每个阶段都有相应的标准、规范和流程，并有相应的提交件。前一阶段未完成之前不能进入下一阶段，从而保证项目有序进行。

### 1.1 标准阶段划分（短周期项目，约90天）

| 阶段 | 工作内容 | 参考工期 |
|------|---------|---------|
| 项目启动 | 合同签订即启动项目 | 第1天 |
| 需求分析 | 需求调研及需求文档编写 | 约20天 |
| 系统设计 | 系统设计及完善、初始化数据收集 | 约8天 |
| 代码开发 | 系统代码开发及完善 | 约50天 |
| 系统测试 | 系统功能测试、系统集成测试 | 约17天（与开发尾声重叠） |
| 系统部署 | 系统部署上线 | 约1天 |
| 系统试运行 | 系统进行正式试调运行 | 约30天 |
| 系统终验 | 完成最终验收 | 约3天 |

### 1.2 完整阶段划分（大型项目，约12个月）

| 阶段 | 工作内容 | 参考工期 |
|------|---------|---------|
| 项目启动与需求基线确立 | 快速明确业务蓝图并交付标准产品，确立需求基线 | 约2个月 |
| 设计蓝图明确 | 所有需求与设计方案的最终确认 | 约2个月 |
| 开发完成与测试完成 | 定制化开发与集成完成、系统测试完成 | 约5个月 |
| 上线试运行与最终验收 | 系统试运行、正式上线、最终验收 | 约3个月 |

## 二、里程碑设置模板

以清晰的关键路径为核心，设置一系列合理的里程碑节点，确保项目在可控的轨道上稳步推进。

### 2.1 里程碑节点模板

| 里程碑编号 | 里程碑名称 | 里程碑标志 | 参考时间点 |
|-----------|-----------|-----------|-----------|
| M1 | 项目启动与需求基线确立 | 第一轮需求评审完成、标准产品部署上线 | 合同签订后第1-2个月 |
| M2 | 设计蓝图明确 | 第二轮需求与设计评审完成 | 合同签订后第3-4个月 |
| M3 | 开发完成 | 定制化开发与集成完成 | 合同签订后第5-7个月 |
| M4 | 测试完成 | 系统测试完成，达到可部署质量标准 | 合同签订后第7-8个月 |
| M5 | 试运行开始 | 系统试运行启动 | 合同签订后第8-9个月 |
| M6 | 正式上线 | 系统正式上线 | 合同签订后第9-10个月 |
| M7 | 最终验收通过 | 项目验收完成，进入质保期 | 合同签订后第10-12个月 |

### 2.2 里程碑缓冲设计

计划的时间安排应充分考虑任务间的依赖关系、复杂度和必要的缓冲时间：
- 优先部署标准产品供试点运行
- 在核心开发结束后设置测试与修复期
- 在试运行后预留用户适应与验收准备时间
- 过程中采用每周功能迭代方式进行开发、测试，以及单独功能试运行

## 三、阶段性交付物模板

每个阶段都有明确的交付物，确保可衡量和可验证。

| 阶段 | 交付物 |
|------|--------|
| 项目启动阶段 | 项目章程、项目计划书、资源计划 |
| 标准产品部署阶段 | 部署完成的管理系统、培训材料、用户手册 |
| 初始化与试用阶段 | 初始化报告、试用反馈报告、优化记录 |
| 需求分析阶段 | 需求规格说明书、评审报告 |
| 系统设计阶段 | 系统设计文档、数据库设计文档 |
| 开发阶段 | 可运行的系统代码、单元测试报告 |
| 测试阶段 | 测试计划、测试用例、测试报告、缺陷修复记录 |
| 试运行阶段 | 试运行报告、用户满意度调查 |
| 验收阶段 | 验收申请、验收报告、项目总结文档 |
| 正式上线 | 上线报告 |
| 质保期 | 维护报告、服务记录 |

## 四、进度管理方法

### 4.1 进度管理原则

通过项目进度管理使全体成员积极主动，在项目进展中遇到问题主动找相关人员解决。若解决不得力而确实属此人管理的，应及时向上一级反映，不得以任何借口推脱不按进度计划完成任务。质量保证组有责任监控和汇报项目的进度状况。

### 4.2 进度控制方法

1. 项目经理必须根据任务情况做好进度安排计划，按周做计划以书面呈交项目协调委员会
2. 以周为单位做计划以书面形式下达各组，各组分头安排落实到个人
3. 组长或个人在接到计划书时，认为恰当则签字；若认为不恰当，必须及时陈述理由
4. 在计划时间到时，项目经理严格按照进度计划书验收
5. 质量保证组作为项目组外的成员，客观地监控和评价项目的进展情况

### 4.3 WBS分解原则

- 项目生命周期的各阶段作为分解的第二层
- 主要可交付物作为分解的第二层
- 整合可能由项目团队以外的组织来实施的各种组件
- 工作包大小至少需要8小时，总完成时间不超过80小时
- WBS不是某个项目成员的责任，应由全体项目团队、用户和项目干系人共同完成

## 五、项目主要基础文档清单

| 序号 | 文档名称 | 说明 |
|------|---------|------|
| 1 | 项目实施计划及方案 | 应用范围、架构、开发前提、项目计划及开发步骤概述 |
| 2 | 业务需求说明书 | 需求、必要条件和约束条件的审核和描述 |
| 3 | 概要设计说明书 | 设计符合需求，与项目范围和目标一致 |
| 4 | 详细设计说明书 | 模块间组成部分的设计和接口 |
| 5 | 数据库设计说明书 | 数据库方面的组成部分 |
| 6 | 外部接口设计文档 | 接口对接实现系统互联的开发组成部分 |
| 7 | 全面的测试计划和测试案例及报告 | 每一测试的策略、组织、方法和资源的描述 |
| 8 | 用户培训手册、操作手册 | 最终用户的操作程序 |
| 9 | 系统安装部署手册 | 系统部署说明 |
| 10 | 项目验收报告 | 验收过程及结果 |
| 11 | 运维手册 | 系统运维操作指南 |
| 12 | 试运行情况报告 | 试运行期间的系统运行情况 |

### 中型项目快速交付方案（约120天）

适用于功能模块较为成熟、以配置和定制为主的投资管理系统项目：

| 阶段 | 工作内容 | 周期 |
|------|---------|------|
| 项目启动 | 合同签订即启动，组建团队，开工会 | Day 1 |
| 需求分析 | 需求调研、业务流程梳理、需求文档编写确认 | Day 1-20 |
| 系统设计 | 系统设计及完善、数据模型设计、初始化数据收集 | Day 20-28 |
| 代码开发 | 系统代码开发及完善、接口开发、数据迁移开发 | Day 28-80 |
| 系统测试 | 功能测试、集成测试、性能测试、用户验收测试 | Day 68-85 |
| 系统部署上线 | 生产环境部署、数据初始化、系统切换 | Day 86 |
| 系统试运行 | 系统试运行、问题修复、用户培训 | Day 87-116 |
| 系统终验 | 终验评审、文档交付、项目总结 | Day 117-120 |

关键说明：需求分析与系统设计之间留有重叠窗口，系统测试与代码开发后期并行执行，确保整体工期可控。
