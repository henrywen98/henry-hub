# 测试方案

## 一、测试总体策略

建立覆盖全生命周期的质量管控体系，以标准化流程保障系统质量。制定分层级测试策略，涵盖单元测试、集成测试、系统测试、性能测试、安全测试及兼容性测试（适配信创/非信创环境），明确各阶段测试范围、用例设计标准、工具选型（如Jmeter用于性能测试、Selenium用于自动化测试），确保测试覆盖核心业务流程与边界场景。

项目团队至少提供如下测试：
- 单元模块测试
- 内部联调集成测试
- 由业务人员参与的功能性测试
- 与其它有关系统进行信息互通和资源共享方面的测试（数据接口可用性测试）
- 系统整体性能和压力测试

## 二、测试管理方案

### 2.1 制定测试计划

1. 测试经理根据本项目上线计划时间制定测试计划
2. 完成的测试计划需经过内部评审，让测试人员了解自己的测试任务及测试进度安排，并得到确认
3. 针对任务安排中有争议的地方，由测试经理进行协调并调整
4. 测试经理将内部确认通过的测试计划邮件发送给项目负责人进行审核

### 2.2 分析测试需求

1. 测试人员根据分配的需求情况，获取原始需求文档和开发组提交的设计方案
2. 对测试需求进行整理，用容易理解的自然语言明确描述一项需要测试的内容，尽可能剥离多项内容，保证一条测试需求只包含一项测试内容，形成《测试方案》初稿
3. 完成的测试方案由测试经理组织进行内部评审（组内交叉评审）
4. 测试人员针对评审发现的问题进行修改
5. 将内审通过的《测试方案》上传到配置管理服务器上

### 2.3 设计测试用例

1. 测试人员针对测试方案中所列的测试案例标题，进行细化，补充预置条件、测试步骤、预期结果等重要信息，形成《测试案例》
2. 提交给测试经理安排项目组内部评审，形成《评审记录》
3. 相关人员修改评审过程中发现的问题，并得到问题提出人的确认
4. 将内部评审通过的《测试案例》提交到配置管理服务器上

### 2.4 测试环境准备

1. 开发组提供完整的部署包，包括所有应用、服务、进程和SQL脚本，以及版本说明文档
2. 环境部署完成后，由测试人员对负责的需求先进行冒烟测试，验证是否存在环境问题
3. 所有冒烟测试过程中发现的问题要在正式测试执行开始前解决

### 2.5 测试执行与缺陷提交

1. 测试人员根据确认的测试案例执行测试，缺陷统一提交到缺陷管理系统中，状态为New
2. 测试经理对提交的缺陷先进行内部审核，审核通过的指派给开发组接口人
3. 被开发组置为"拒绝"状态的缺陷，若测试人员认为仍是缺陷，可与项目负责人确认后重新打开
4. 状态为"延期"的缺陷，需告知项目负责人并得到其认可
5. 状态为"修复"的缺陷进行验证测试，确认通过置为"关闭"，仍有问题则"重新打开"
6. 测试经理每天将测试进度导出，以邮件形式发送给项目负责人

### 2.6 编写测试报告

1. 上线前一天，测试人员将测试结果进行汇总，完成测试报告
2. 各自报告发给测试经理审核并整合
3. 最终的验收测试报告以邮件形式发送给项目负责人，并提交到配置管理服务器上

### 2.7 各阶段工作成果

| 阶段 | 工作成果 |
|------|---------|
| 制定测试计划 | 《测试计划》《测试计划评审记录》 |
| 分析测试需求 | 《测试方案》《测试方案评审记录》 |
| 编写测试用例 | 《测试案例》《测试案例评审记录》 |
| 测试环境准备 | 《部署问题记录》《冒烟测试结果》 |
| 测试执行与缺陷提交 | 《测试记录》《缺陷列表》完善的《测试用例》《进度记录》 |
| 编写测试报告 | 《测试报告》《测试报告评审记录》 |

## 三、自动化测试

### 3.1 自动化测试目的

为保证新版本的质量，每次上线前都需要经历严格的测试，包括对新增功能或修改功能的测试，以及对原有相关功能和核心功能的回归测试。自动化功能测试是提高回归测试质量和效率的理想解决方案，为已通过手工测试验证、功能比较稳定的模块编写自动化测试脚本。

### 3.2 自动化测试方法

采用模块化的方法，为应用软件中的每个基本功能或每个画面创建一个脚本或文件，设置若干检查点和同步点，将所有数据参数化，对外部数据文件采取引用方式读取。在脚本中内置逻辑，对外部数据文件、对象、数据点进行检查。如果软件功能或画面发生改变，只需更新相应的基本功能脚本，就能自动修复所有调用该功能的测试用例。

### 3.3 自动化测试流程（五阶段）

依照软件工程学原理，将整个自动化流程分为五个阶段：

1. **准备阶段**：确定自动化工具，搭建自动测试脚本制作平台；确定常规运行模式，搭建自动测试运行平台
2. **用例设计/初建框架阶段**：设计自动化测试框架
3. **建立系统重要功能自动测试阶段**：主要解决最低测试质量保证问题，从手工测试用例库中挑选重要的或常用的功能模块，作为实施自动化测试的切入点
4. **建立核心业务集自动回归测试阶段**：进一步扩大自动化测试覆盖程度，使用自动化测试框架完成核心业务流程的测试
5. **建立标准自动回归测试阶段**：再次扩大自动化测试的覆盖程度，将可以用自动化测试实现的功能尽量自动化，确保软件版本的完整质量

## 四、性能测试

### 4.1 性能测试目标

通过自动化测试工具模拟多种正常、峰值以及异常负载条件来对系统的各项性能指标进行测试。客观地测试系统的各项性能指标，对系统的稳定性、性能表现做出评价；对存在性能瓶颈的产品提出性能优化建议和方案。

### 4.2 性能测试方法

| 测试类型 | 说明 |
|---------|------|
| 负载测试 | 模拟指定并发用户数的负载压力，记录事务平均响应时间、成功率等指标 |
| 压力测试 | 不断施加压力直至系统响应超时或出错，确定系统的瓶颈或不能接收请求的性能点 |
| 强度测试 | 确定系统稳定运行下能支持的最大并发用户数，持续执行并综合分析各项指标 |
| 容量测试 | 测试大数据量下的系统性能表现，或确定给定环境下系统所能承受的数据量极限 |

### 4.3 性能测试流程

**前置条件**：功能已通过测试且关键功能和业务稳定；软件已置于配置管理之下；各种技术文档完备。

**测试实施步骤**：

1. **项目需求分析**：定义测试范围、确定压力值；确定进度计划；明确各方人员职责分工。需采集的数据包括：
   - 系统前台终端的分布情况
   - 系统忙时、闲时典型业务的业务量、响应时间
   - 预测系统未来3-5年用户量增长趋势
   - 系统硬件配置容量

2. **测试设计阶段**：
   - 测试脚本准备（通过LoadRunner录制或手工编写）
   - 设计性能测试场景（单场景和综合场景）
   - 规划测试数据的存储和管理规范

3. **测试执行阶段**：先进行小并发量的预测试，正式测试按定义的测试范围、场景、迭代次数完成执行。测试执行原则：
   - 性能测试脚本进行参数化
   - 所有虚拟用户以线程方式运行，均匀分布在所有负载模拟客户端
   - 测试环境在同一时间内只执行一个测试用例
   - 同一测试用例需执行两次以上，取平均值

4. **系统性能监控**：
   - 安装应用服务器监控器（HP Diagnostic监控JVM、Java堆、SQL执行效率等）
   - 监控系统资源（CPU利用率、内存占用率、磁盘I/O使用率）
   - 监控应用服务器（JDBC连接池、等待队列、Java堆使用情况）
   - 监控数据库（连接数、缓冲池使用、死锁等指标）
   - 监控网络吞吐量

5. **测试总结阶段**：输出文档包括性能测试方案、测试用例、脚本文件、测试场景、测试结果记录、性能瓶颈分析和调优建议书、最终性能测试报告

## 五、缺陷管理与闭环机制

构建"识别-上报-处理-验证-归档"的全流程缺陷管理规范。采用禅道项目管理软件：

- **缺陷识别阶段**：通过测试执行、用户反馈等多渠道收集问题，统一记录缺陷类型、严重程度（致命/严重/一般/轻微）、复现步骤等关键信息
- **缺陷处理阶段**：按严重程度分级流转，明确责任部门与整改时限
- **闭环机制**：建立缺陷跟踪台账，修复后经测试团队验证通过方可闭环，未通过则返回重新处理，同时定期开展缺陷分析

### 缺陷状态流转

缺陷状态流转：New（新提交）→ 指派（分配给开发人员）→ 修复/拒绝/延期 → 验证 → 重新打开/关闭。测试人员每天更新需求测试进展。缺陷按严重程度分为：致命（系统崩溃/数据丢失）、严重（核心功能不可用）、一般（次要功能异常）、轻微（界面/体验问题）。致命和严重缺陷需24小时内响应修复。

### 自动化测试建设路径

分五阶段逐步建设：
1. **搭建平台**：搭建自动化测试环境和框架（Selenium/Playwright + TestNG/JUnit）
2. **初建框架**：建立自动化测试代码规范、公共组件库和数据驱动模板
3. **重要功能自动测试**：对核心功能（登录、审批流程、报表生成）编写自动化用例
4. **核心业务集回归**：将核心业务流程（投资全流程、基金全生命周期）纳入回归测试集
5. **标准回归**：建立完整的回归测试集，每次版本发布前自动执行

采用模块化方法为每个基本功能创建独立脚本，通过组合脚本实现端到端业务流程测试。

### 性能测试详细方案

使用LoadRunner/JMeter进行性能测试，监控维度包括：应用服务器性能（APM工具）、系统资源（CPU/内存/磁盘I/O/网络）、数据库（慢查询/连接池/锁等待）、网络（带宽/延迟/丢包率）。测试类型补充：
- **强度测试**：在资源受限条件下（如CPU占用80%、内存占用90%）验证系统稳定性
- **容量测试**：验证系统可承载的最大数据量和最大并发用户数
