# 输入

用户提供招标文件路径（支持 `.docx`、`.doc`、`.pdf` 格式）。

如果用户未提供路径，提示用户输入招标文件路径后再继续。

# 文件读取策略

根据文件扩展名选择对应的读取方式：

1. **`.docx` 文件**：使用 `/docx` skill 读取（来自 henry-hub）
2. **`.doc` 文件**（旧版 Word 二进制格式）：先用 macOS 内置工具转换为 .docx，再用 `/docx` skill 读取
   ```bash
   textutil -convert docx "<原始文件路径>"
   ```
   转换后的 .docx 文件会出现在同一目录下，文件名相同但扩展名变为 .docx。
3. **`.pdf` 文件**：使用 `/pdf` skill 读取

读取完成后，确保获取到招标文件的完整文本内容，包括正文和所有表格数据。

# 执行步骤

## 步骤1：读取招标文件

按上述文件读取策略加载招标文件的全部内容。如果文件很大，分段读取确保不遗漏。

## 步骤2：提取所有表格

从文件内容中识别并提取以下类型的表格：

- **评分表**（技术评分表、商务评分表）：包含评分大项、评分细项、分值、评分标准
- **需求表**（功能需求表、技术需求表）：包含序号、模块名称、功能点、核心说明/要求描述
- **报价表**：分项报价模板（如有）
- **资格要求表**：投标人资格条件汇总（如有）
- **其他表格**：服务要求表、人员要求表等

对每张表格，保留其完整结构（行列关系），标注表格在文件中的位置（所属章节）。

## 步骤3：提取章节结构和关键段落

识别招标文件的层级结构：
- 一级章节（如：采购需求、评分办法、商务条件、资格要求）
- 二级/三级章节
- 各章节下的关键段落（工期要求、服务要求、技术要求等具体条款）

重点提取以下内容：
- **基本信息**：采购人名称、项目名称、投标截止日期、工期要求、预算金额
- **评分体系**：所有评分项的大项、细项、分值、评分要点
- **业务功能需求**：需求表中的每一行（模块名称 + 功能点 + 要求描述）
- **技术需求**：技术架构、安全、性能等非功能性要求
- **服务要求**：工期、维保、培训、人员配置等
- **资格要求**：资质证明、案例要求（数量、时间范围、行业）、人员资质
- **商务要求**：报价方式、付款节点、合同条款
- **关键术语**：扫描全文，提取项目名词（项目全称、采购人、系统名称）、高频业务术语（出现≥3次的业务概念）、技术要求术语（技术需求中明确提及的技术词汇），按类别整理为术语清单

## 步骤4：知识库匹配

读取知识库索引文件：
```
${CLAUDE_PLUGIN_ROOT}/knowledge/index.md
```

对步骤2和步骤3中提取的**每一个业务功能需求点**，使用语义理解进行匹配：

1. 阅读 `index.md` 中每个条目的文件路径和内容摘要
2. 对招标文件中的每个功能需求点，判断其语义上是否与某个知识库条目对应
   - 不要做简单的关键词匹配，而是理解需求点的**业务含义**
   - 例如："项目全生命周期管理" 应匹配 "股权投资管理"（投前-投中-投后）
   - 例如："数据可视化大屏" 应匹配 "数据驾驶舱与报表"
   - 例如："统一身份认证对接" 应匹配 "系统集成"
3. 为每个需求点标注匹配结果：
   - **已匹配**：标注匹配到的知识库文件路径
   - **部分匹配**：标注最相关的知识库文件路径，并说明差异点
   - **未匹配**：标记为"知识库未覆盖，需人工补充"

## 步骤5：生成结构化解析报告

按照下方输出模板生成完整的解析报告。

## 步骤6：持久化保存

将解析报告保存到文件：
```
./bid-output/parse-report.md
```

如果 `./bid-output/` 目录不存在，先创建该目录。

## 步骤7：用户确认

报告输出并保存后，向用户展示以下提示：

> 以上解析是否准确？如需调整，请直接指出需要修改的部分。确认无误后运行 `/bid.plan` 生成标书骨架。

# 输出模板

按 `${CLAUDE_PLUGIN_ROOT}/skills/parse/references/output-template.md` 中的格式输出解析报告。

# 注意事项

1. 评分体系是最关键的信息，务必完整提取每一个评分项和分值，不要遗漏
2. 需求清单要逐条提取，保持与原文的对应关系
3. 知识库匹配使用语义理解，不要依赖简单的关键词命中
4. 对于招标文件中模糊或有歧义的内容，在解析报告中如实标注，不要自行推测
5. 如果招标文件中某个章节的内容在解析报告模板中没有对应位置，将其归入"其他风险提示"
6. 金额、日期等关键数据必须与原文完全一致，不做任何转换或近似处理
