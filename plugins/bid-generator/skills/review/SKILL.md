# 标书审查 — 五维质量评估

以标书评审专家视角执行五维质量审查（评分覆盖度、需求对应、术语复现、待更新项、篇幅合理性），确保审查结论客观、可操作。

---

## 输入说明

审查对象为 `./bid-output/` 目录下的已生成文件：

```
./bid-output/
├── parse-report.md          # /bid.parse 的解析报告（评分体系 + 需求清单）
├── plan.md                  # /bid.plan 的骨架（章节结构 + 篇幅权重）
├── sections/                # /bid.fill 生成的各章节正文
│   ├── *.md
│   └── ...
└── review-report.md         # 本 skill 的输出（审查报告）
```

---

## 步骤一：收集审查材料

### 目标

加载全部投标文件和参考基准，为五维审查做准备。

### 执行

1. **读取解析报告** — 读取 `./bid-output/parse-report.md`，提取：
   - 评分体系（评分项 + 分值 + 评分要点）
   - 业务功能需求清单（功能点 + 要求摘要）
   - 技术需求清单
   - 非功能需求清单
   - 服务要求摘要
   - 资格要求摘要
   - 招标文件中的关键术语（项目名称、业务术语、技术术语）

2. **读取骨架计划** — 读取 `./bid-output/plan.md`，提取：
   - 完整章节结构（一级/二级/三级标题）
   - 每个章节的篇幅权重（星级）和参考字数
   - 知识库来源标注
   - 待更新标记（⚠️）

3. **读取所有已生成章节** — 遍历 `./bid-output/sections/` 目录，读取全部 `.md` 文件，记录：
   - 文件名和对应章节名
   - 实际字数（不含 Markdown 标记和 YAML frontmatter）
   - 文件内容全文（供后续检索）

4. **状态检查** — 若 `parse-report.md` 或 `plan.md` 不存在，提示用户先运行 `/bid.parse` 和 `/bid.plan`。若 `sections/` 目录为空或不存在，提示用户先运行 `/bid.fill` 填充内容。

**确认材料加载完毕后，进入五维审查。**

---

## 步骤二：覆盖度检查（vs 评分体系）

### 目标

逐一核对评分表中的每个评分项是否有对应的投标文件章节，评估总覆盖率。

### 执行

1. 从解析报告中提取完整评分表，列出全部评分大项和评分细项
2. 对每个评分细项，在 `sections/` 目录的文件中搜索对应章节：
   - 先按章节标题精确匹配
   - 若无精确匹配，按评分要点关键词在正文中模糊搜索
3. 逐条标记覆盖状态：
   - **已覆盖** — 有对应章节且内容实质性回应了评分要点
   - **部分覆盖** — 有对应章节但未完整回应全部评分要点
   - **未覆盖** — 无对应章节或内容为空/占位符
4. 计算总覆盖率 = 已覆盖数 / 评分项总数 x 100%
5. 计算加权覆盖率 = 已覆盖评分项的分值总和 / 评分总分 x 100%（反映分值维度的覆盖情况）

### 输出表格

按 `${CLAUDE_PLUGIN_ROOT}/skills/review/references/check-templates.md` 中"表1：评分覆盖度检查"的格式输出。

## 步骤三：需求逐条对应（vs 需求清单）

### 目标

检查解析报告中的每一条需求是否在投标文件中有明确的回应位置，确保无遗漏。

### 执行

1. 从解析报告中提取全部需求条目，包括：
   - 业务功能需求（功能点名称 + 核心说明）
   - 技术需求
   - 非功能需求（性能、安全、可用性等）
   - 服务要求（工期、维保、培训等）
2. 对每条需求，在 `sections/` 的文件中定位其回应位置：
   - 搜索需求关键词在正文中的出现位置
   - 记录所在文件名和章节标题
3. 逐条标记对应状态：
   - **已对应** — 在投标文件中找到明确的回应段落
   - **弱对应** — 有提及但未针对性回应（仅一笔带过）
   - **未对应** — 在投标文件中未找到任何回应
4. 计算需求覆盖率 = 已对应数 / 需求总数 x 100%

### 输出表格

按 `${CLAUDE_PLUGIN_ROOT}/skills/review/references/check-templates.md` 中"表2：需求逐条对应检查"的格式输出。

## 步骤四：关键术语复现检查

### 目标

检查招标文件中的关键术语在投标文件中的出现频次，确保投标文件"说招标方的语言"。

### 执行

1. 从 `./bid-output/parse-report.md` 的「一-A、关键术语清单」段落中直接读取术语列表。若该段落不存在（旧版 parse 报告），则从解析报告全文中推断关键术语，分为三类：
   - **项目专有名词** — 项目名称、采购人名称、系统名称等
   - **业务术语** — 招标文件中反复出现的业务概念（如"股权投资"、"基金管理"、"风控合规"、"全生命周期"等）
   - **技术术语** — 招标文件中明确提及的技术要求术语（如"微服务"、"信创"、"等保三级"等）
2. 在 `sections/` 全部文件的正文中搜索每个术语的出现次数
3. 按频次判定复现状态：
   - **充分复现** — 出现 3 次及以上
   - **有限复现** — 出现 1-2 次
   - **未复现** — 出现 0 次
4. 重点关注"未复现"的术语 — 这些是评审专家可能注意到的遗漏

### 输出表格

按 `${CLAUDE_PLUGIN_ROOT}/skills/review/references/check-templates.md` 中"表3：关键术语复现检查"的格式输出。

## 步骤五：待更新项检查

### 目标

搜索投标文件中所有标记为"需更新"的占位内容，确认用户已用真实信息替换。

### 执行

1. 在 `sections/` 全部文件中搜索以下占位标记：
   - HTML 注释占位符（/bid.fill 生成格式）：`<!-- TODO: [需更新]`、`<!-- TODO: [需补充]`
   - Markdown 标记：`⚠️需更新`、`⚠️需填写`、`⚠️需人工补充`、`⚠️需人工确认`
   - 通用占位符：`[待填写]`、`[请补充]`、`[TODO]`、`[TBD]`、`[xxx]`
2. 记录每个占位标记的：
   - 所在文件名
   - 所在章节标题
   - 占位内容摘要（上下文 20 字以内）
   - 内容类别（公司信息 / 案例信息 / 团队信息 / 报价金额 / 其他）
3. 统计各类别的占位数量

### 输出表格

按 `${CLAUDE_PLUGIN_ROOT}/skills/review/references/check-templates.md` 中"表4：待更新项检查"的格式输出。

## 步骤六：篇幅合理性检查

### 目标

对比各章节的实际字数与骨架中标注的参考字数，检查篇幅分配是否合理。

### 执行

1. 从 `plan.md` 中提取每个章节的：
   - 章节名称
   - 篇幅权重（星级：★ ~ ★★★★★）
   - 对应分值区间
2. 根据篇幅权重确定参考字数范围（取中间值作为参考基准）：
   - ★★★★★（≥20 分）→ 参考 12500 字（10000-15000）
   - ★★★（10-19 分）→ 参考 6500 字（5000-8000）
   - ★★（5-9 分）→ 参考 3500 字（2000-5000）
   - ★（<5 分）→ 参考 1500 字（1000-2000）
3. 统计 `sections/` 中每个文件的实际字数（不含 Markdown 标记和 YAML frontmatter）
4. 计算偏差率 = (实际字数 - 参考字数) / 参考字数 x 100%
5. 按偏差率判定篇幅状态：
   - **合理** — 偏差率在 -30% ~ +50% 之间
   - **偏少** — 偏差率 < -30%（内容可能不够充分）
   - **偏多** — 偏差率 > +50%（内容可能冗余或跑题）
   - **严重不足** — 偏差率 < -50%（需重点关注）

### 输出表格

按 `${CLAUDE_PLUGIN_ROOT}/skills/review/references/check-templates.md` 中"表5：篇幅合理性检查"的格式输出。

## 步骤七：生成审查报告

### 目标

汇总五维审查结果，输出完整审查报告并给出改进建议。

### 报告模板

按 `${CLAUDE_PLUGIN_ROOT}/skills/review/references/check-templates.md` 中"最终审查报告模板"的格式，将五维审查实际数据填入模板，保存到 `./bid-output/review-report.md`。

### 执行

1. 按上述模板填入五维审查的实际数据
2. 判定总体状态：
   - **通过** — 5 个维度全部为 ✓
   - **需改进** — 存在 ⚠️ 但无 ✗
   - **需大幅修改** — 存在任何 ✗
3. 撰写针对性的改进建议，按优先级分三档，每条建议须具体到章节和操作
4. 对于以下可自动修复的问题，在改进建议中提供具体的修复指令：
   - **未复现的"必现"术语**：指出需要在哪些章节的哪个位置插入该术语，给出建议的句子片段
   - **篇幅严重不足**：指出需要扩展的子章节和建议扩展方向
   提示用户可选择让 Claude 直接执行修复，或手动处理。
5. 保存报告到 `./bid-output/review-report.md`
6. 在对话中向用户展示审查总览表格和高优先级改进建议
7. 提示用户："审查报告已保存到 `./bid-output/review-report.md`。建议优先处理高优先级问题，修改后可再次运行 `/bid.review` 复查。"
