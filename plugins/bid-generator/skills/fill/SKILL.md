# /bid.fill — 按需填充内容

## 输入

- **单章模式**：`/bid.fill <章节名>` — 填充指定章节（章节名需与骨架 plan.md 中的条目匹配）
- **批量模式**：`/bid.fill all` — 按骨架顺序逐章填充全部章节

## 执行步骤

### 步骤1：读取骨架，定位章节

读取 `./bid-output/plan.md`，从中获取目标章节的：
- 章节标题与层级
- 评分分值与篇幅权重（星级）
- 关联的知识库文件路径
- 招标需求原文要点
- 是否标记为需人工补充或需更新

如果 `./bid-output/plan.md` 不存在，提示用户先运行 `/bid.plan` 生成骨架。

### 步骤2：确定写作模式

读取 `${CLAUDE_PLUGIN_ROOT}/rules/writing-patterns.md`，根据章节类型选择对应的写作模式（模式1-7）。

**章节类型到写作模式的映射表：**

| 章节类型 | 写作模式 | 适用章节举例 |
|---------|---------|------------|
| 架构设计 | 模式1：架构设计类 | 总体架构设计、技术架构设计、数据架构设计、业务逻辑架构设计、部署架构设计 |
| 功能模块 | 模式2：功能模块类 | 股权投资管理、基金管理、风控合规、监管报送、数据驾驶舱与报表、统一门户与工作台、文档与档案管理、系统集成、AI智能能力、相关方管理、委托贷款、保理业务、租赁业务 |
| 实施方案 | 模式3：实施方案类 | 项目组织与沟通、实施方法论、进度与里程碑、测试方案、验收方案、规范性保障、风险管控 |
| 维保方案 | 模式4：维保方案类 | 服务范围与内容、定期巡检计划、故障处理与响应、质量保证措施、应急预案、维保团队 |
| 培训售后 | 模式5：培训方案类 | 培训方案、售后服务、知识转移 |
| 公司案例 | 模式6：公司与案例类 | 公司资质、项目案例、核心团队 |
| 偏离表 | 模式7：偏离表类 | 技术偏离表、商务偏离表 |
| 需求分析 | 模式1（简化） | 项目背景与现状、建设目标、建设内容概述 |
| 商务报价 | 模式7（变体） | 报价方法说明、分项报价模板 |
| 程序文件 | 模式7（变体） | 投标书模板、授权委托书模板、声明函模板 |

> 若章节类型不在上表中，根据章节内容就近匹配最合适的模式。

### 步骤3：读取知识库

读取 `${CLAUDE_PLUGIN_ROOT}/knowledge/` 目录下骨架中标注的对应知识库文件。

- 先读取 `${CLAUDE_PLUGIN_ROOT}/knowledge/index.md` 定位文件路径
- 再读取具体的知识库 .md 文件，提取正文内容作为生成素材
- 注意文件 frontmatter 中的 `needs_update` 字段，若为 `true` 则在输出中标记需更新

### 步骤4：生成正文

综合以下三个输入生成该章节的完整 Markdown 正文：

1. **写作模式**（步骤2）— 决定章节结构和语气
2. **知识库内容**（步骤3）— 提供专业素材和表述基础
3. **招标需求要点**（步骤1从骨架中获取）— 确保内容精准应标

生成时遵循以下规则：

- **术语复现**：招标文件中的关键术语（功能名称、技术要求、指标参数等）必须在正文中原文复现，不得改写或省略
- **核心术语强制复现**：从 `./bid-output/parse-report.md` 的「关键术语清单」中提取所有标记为"必现"的术语（项目全称、采购人名称、系统名称），确保：
  - 每个章节的开篇概述中至少出现1次项目全称或系统名称
  - 每个一级章节中"高频"业务术语至少出现2次
  - 生成完成后自查：若某个"必现"术语在全文中出现0次，在生成说明中标记为⚠️警告
- **架构图规范**：技术类章节（架构设计类、系统需求类）应配专业架构图：
  - 每个一级技术章节配1张总体架构图
  - 使用 ASCII art 文本框图格式（确保 Markdown 渲染兼容）
  - 架构图须标注：层级名称、核心组件名称、数据流方向、关键技术选型
  - 若骨架中某章节标注了"📊 架构图"，必须生成对应的图
- **篇幅控制**：根据评分分值权重控制篇幅深度
  - 5星（>=20分）：10000-15000字
  - 3星（10-19分）：5000-8000字
  - 2星（5-9分）：2000-5000字
  - 1星（<5分）：1000-2000字
- **需更新内容占位**：知识库中标记 `needs_update: true` 的内容（公司资质、项目案例、核心团队等），使用 HTML 注释占位符标记：
  ```markdown
  <!-- TODO: [需更新] 请替换为贵公司实际的xxx信息 -->
  ```
- **需人工补充内容**：知识库未覆盖的需求点，同样使用占位符：
  ```markdown
  <!-- TODO: [需补充] 知识库未覆盖此需求点，请人工补充xxx内容 -->
  ```
- **不凭空编造**：数据、案例、资质等事实性内容只从知识库中提取，不凭空生成

### 步骤5：输出正文与生成说明

输出两部分内容：

**第一部分：章节正文**（完整 Markdown）

**第二部分：生成说明**（附在正文末尾，用分隔线隔开）

```markdown
---

> **生成说明**
> - 写作模式：模式X（xxx类）
> - 知识库来源：`knowledge/xx-xxx/xxx.md`
> - 应标点覆盖：已覆盖 X/Y 个招标需求要点
> - 未覆盖要点：（列出未覆盖的要点，如有）
> - 需人工处理：（列出需更新/需补充的项，如有）
```

## 持久化

将生成的章节正文（含生成说明）保存到文件：

```
./bid-output/sections/<章节名>.md
```

- 如果 `./bid-output/sections/` 目录不存在，自动创建
- 保存后输出确认信息：`已保存到 ./bid-output/sections/<章节名>.md`

## 批量模式（/bid.fill all）

当输入为 `all` 时，按以下逻辑执行：

1. 读取 `./bid-output/plan.md` 获取全部章节的有序列表
2. 逐章循环调用填充流程（步骤1-5）
3. **跳过已有**：如果 `./bid-output/sections/<章节名>.md` 已存在，跳过该章节并输出提示：`[跳过] <章节名> — 文件已存在`
4. **断点续填**：若中途中断，再次运行 `/bid.fill all` 会从第一个未生成的章节继续，已有文件不会被覆盖
5. 每完成一个章节后输出进度：`[已完成 X/Y] <章节名>`
6. 全部完成后输出汇总：已填充章节数、跳过章节数、需人工处理项汇总
