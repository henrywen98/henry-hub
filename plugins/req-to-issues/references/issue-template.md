# Issue 模板与编写规则

## Issue 模板

```markdown
## 需求描述

[用业务语言描述这个功能点]
- 做什么：[用户可以执行的操作或系统提供的能力]
- 为谁做：[目标用户角色]
- 解决什么问题：[当前的痛点或业务场景]

## 业务价值

[为什么要做这个功能]
- 对用户：[用户获得什么好处]
- 对业务：[业务获得什么价值]
- 不做会怎样：[不做的风险或代价]

## 验收标准

- [ ] [标准1]
- [ ] [标准2]
- [ ] ...

## 影响范围

- 涉及模块：[模块列表]
- 涉及角色：[用户角色列表]

## 依赖关系

- 前置：#issue_number [描述]
- 被依赖：#issue_number [描述]

## 约束条件

[非功能需求，如无则写"无"]

## 优先级

P0/P1/P2/P3

## 标签

feature / enhancement / bug / refactor
```

## 编写规则

### 需求描述

- **使用业务语言**，避免技术术语（不写"API"、"数据库"、"接口"等）
- 描述用户可感知的行为变化，不描述内部实现
- 一个 Issue 对应一个独立可交付的功能点
- 功能点的判断标准：能单独演示给用户看、能独立测试、能独立发布

错误示例：
> 创建一个 RESTful API，包含 GET/POST/PUT/DELETE 端点，使用 JWT 认证

正确示例：
> 用户可以在订单列表页查看所有销售订单，支持按状态筛选和按日期排序

### 业务价值

- 必须回答"为什么要做"而非"做什么"
- 尽量量化（提升效率 X%、减少 Y 步操作、覆盖 Z 个场景）
- 如果无法量化，描述定性价值（降低出错率、提升用户体验）

### 验收标准

- 每条必须是**客观可判定**的（能回答"是/否通过"）
- 覆盖**正向流程**和**关键异常流程**
- 使用具体的数据和场景，避免抽象描述
- 数量建议：3-8 条，不宜过多也不宜过少

错误示例：
> - [ ] 系统应该好用
> - [ ] 页面加载速度快

正确示例：
> - [ ] 用户输入客户名称关键词后，列表在 1 秒内显示匹配结果
> - [ ] 当订单金额为 0 时，系统提示"订单金额不能为空"且无法提交

### 优先级定义

| 级别 | 含义 | 判断标准 |
|------|------|---------|
| P0 | 必须做 | 不做则核心流程跑不通、或有合规/安全风险 |
| P1 | 应该做 | 不做会明显影响用户体验或业务效率 |
| P2 | 可以做 | 做了更好，不做也能接受 |
| P3 | 将来再说 | 好主意但不紧急，先记录 |

### 标签规则

| 标签 | 含义 |
|------|------|
| `feature` | 全新功能，之前完全没有 |
| `enhancement` | 已有功能的改进或扩展 |
| `bug` | 现有功能的缺陷修复 |
| `refactor` | 不改变外部行为的内部重构 |
| `docs` | 文档相关 |

### 依赖关系

- 只标注**强依赖**（必须先完成 A 才能开始 B）
- 弱依赖（先做 A 更好但不是必须）不标注，避免过度约束开发顺序
- 同批次 Issue 之间用序号引用，上传后会自动替换为 issue number

### 约束条件

常见约束类型：
- **性能**：响应时间、并发量、数据量级
- **兼容性**：浏览器、设备、系统版本
- **安全性**：权限控制、数据加密、审计日志
- **数据**：数据迁移、历史数据兼容
- 如果没有特殊约束，写"无"即可
