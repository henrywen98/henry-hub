# 需求模糊点扫描维度

对需求素材进行 6 个维度的扫描，识别模糊、矛盾或缺失的信息。每个维度列出了具体的检查项，帮助发现需要澄清的问题。

## 1. 业务边界

需求的"做"和"不做"是否清晰。

检查项：
- 功能的起止点在哪里？（从哪个操作开始，到哪个结果结束）
- 哪些场景明确在范围内？哪些场景明确不在范围内？
- 与已有功能的关系：是替代、扩展、还是共存？
- 是否涉及跨系统交互？边界在哪里？

典型模糊信号：
- "等功能"、"之类的"、"相关的" — 范围未闭合
- "优化"、"改进" — 没有说从什么到什么
- 只描述了正向流程，没提到异常场景

## 2. 用户角色

谁会使用这个功能，不同角色的行为是否有差异。

检查项：
- 涉及哪些用户角色？（管理员、普通用户、审批人等）
- 不同角色看到的内容和可执行的操作是否一样？
- 是否有权限差异？（谁能看、谁能改、谁能删）
- 是否涉及审批流程？审批人是谁？

典型模糊信号：
- 只说"用户"不区分角色
- 没有提到权限控制
- 审批流程中的角色不明确

## 3. 功能范围

每个功能点的具体行为是否描述完整。

检查项：
- 用户的操作路径是否完整？（入口 → 操作 → 结果 → 反馈）
- 有哪些必填项？有哪些可选项？
- 数据的增删改查覆盖了哪些？是否需要批量操作？
- 列表页是否需要筛选、排序、分页、导出？
- 是否有状态流转？状态之间的转换条件是什么？

典型模糊信号：
- 只描述了"新增"但没提"编辑"和"删除"
- 有列表但没说筛选条件
- 提到状态但没有状态流转图

## 4. 数据流向

数据从哪来、到哪去、怎么变。

检查项：
- 数据的来源是什么？（用户手动录入、系统自动生成、外部导入）
- 数据创建后会被哪些其他功能引用？
- 数据修改后，引用方是否需要同步更新？
- 是否需要数据导入/导出？格式是什么？
- 历史数据如何处理？

典型模糊信号：
- 只提到"录入信息"但没说信息项
- 模块之间的数据传递关系不明确
- 没有说数据的生命周期（创建→使用→归档/删除）

## 5. 异常场景

非正常路径下系统应如何表现。

检查项：
- 必填项为空时怎么办？
- 数据格式错误时怎么办？
- 重复数据如何处理？
- 并发操作冲突如何处理？（两人同时编辑同一条记录）
- 关联数据被删除后，引用方怎么办？
- 操作是否需要可撤销/可回退？

典型模糊信号：
- 只描述了"happy path"
- 没有提到错误提示信息
- 没有说删除操作是否可逆

## 6. 约束条件

非功能性的限制和要求。

检查项：
- 是否有性能要求？（页面加载时间、数据量上限）
- 是否需要支持移动端？
- 是否有国际化/多语言需求？
- 是否有数据安全和合规要求？
- 是否有与已有系统的集成要求？
- 是否有上线时间要求或里程碑节点？

典型模糊信号：
- 没有提到任何非功能需求（可能是默认、也可能是遗漏）
- "要快"、"要安全" — 没有量化标准

## 提问规则

1. **每轮最多 3 个问题**，避免一次性抛出太多问题让用户疲惫
2. **优先问高影响问题**：影响 Issue 拆分方式的问题优先于细节问题
3. **附带推荐选项**：基于常见做法给出 2-3 个选项，减少用户思考负担
4. **说明提问理由**：简短解释为什么需要澄清这个点（"这会影响是否需要拆成两个独立 Issue"）
5. **允许跳过**：如果用户说"先不管这个"或"以后再说"，记录为开放问题并继续

## 提问格式示例

```
我在分析素材时发现几个需要确认的点：

**Q1: 订单编辑的权限范围**
素材中提到"用户可以编辑订单"，需要确认：
- A) 所有用户都能编辑任何订单
- B) 只能编辑自己创建的订单
- C) 需要特定角色权限才能编辑（推荐，符合常见业务管控模式）
- D) 其他

**Q2: 删除订单的行为**
素材中没有提到删除逻辑，需要确认：
- A) 允许物理删除（从数据库真删除）
- B) 软删除/标记删除（推荐，便于审计追溯）
- C) 不允许删除，只能作废

**Q3: ...**
```
