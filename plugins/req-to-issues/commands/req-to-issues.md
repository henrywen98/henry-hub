## 用户输入

```text
$ARGUMENTS
```

如果为空或 `$ARGUMENTS` 显示为字面值，提示用户在 `/req-to-issues` 后面粘贴需求素材（支持会议纪要、聊天记录、口头描述、变更需求等任何形式）。

## 任务

作为资深业务分析师，将用户提供的原始需求素材转化为 Story 级别的 GitHub/GitLab Issue。每个 Issue 应该是一个独立可交付的功能点，粒度适合开发拿到后使用 `brainstorming` skill 进行技术方案设计。

核心原则：**Issue 严格停留在"需求"层**——使用业务语言，不包含模块划分、技术栈、API 形态、数据库结构等实现细节。那些属于下游 `brainstorming` 阶段的产物。检验标准——"产品经理看得懂吗？开发拿到后知道要做什么吗？"

## 执行步骤

### 1. 加载参考资料

读取以下文件：
- `${CLAUDE_PLUGIN_ROOT}/references/issue-template.md` — Issue 模板和编写规则
- `${CLAUDE_PLUGIN_ROOT}/references/clarify-dimensions.md` — 模糊点扫描维度

### 2. 解析素材

分析用户输入，提取关键信息：
- **业务目标**：这批需求整体要解决什么问题
- **涉及角色**：哪些用户角色会受影响
- **核心功能点**：初步识别有哪些独立的功能点

向用户展示解析结果，确认理解是否正确。

### 3. 识别模糊点 & 逐一澄清

按 clarify-dimensions.md 中的 6 个维度扫描需求素材，找出模糊、矛盾或缺失的信息。

交互规则：
- 每轮最多提 **3 个问题**
- 每个问题附带 **推荐选项**（多选或开放式）和推荐理由
- 用户回答后继续扫描，直到无明显模糊点
- 如果素材已足够清晰，跳过此步骤并告知用户

### 4. 拆解总览

展示拆解方案：
```
共 N 个 Issue：
1. [标题] — [一句话描述]
2. [标题] — [一句话描述]
...
```

标注 Issue 之间的依赖关系（如有）。

询问用户：
- 粒度是否合适（太粗/太细）
- 是否需要合并或拆开某些 Issue
- 是否需要调整优先级排序

根据反馈调整后再次确认。

### 5. 生成 Issue 详情

按 issue-template.md 中的模板，为每个 Issue 生成完整内容：
- 需求描述（业务语言，不含技术细节）
- 业务价值
- 验收标准（checkbox 格式，每条可客观判定是/否）
- 依赖关系（引用同批次其他 Issue 的序号）
- 约束条件
- 优先级（P0-P3）
- 标签

展示所有 Issue 的完整内容供用户最终预览。

### 6. 批量上传

确定目标仓库：
- 如果当前目录是 git 仓库且有 remote，自动检测并确认
- 否则询问用户目标仓库（格式：`owner/repo`）

检测可用的 CLI 工具：
- 优先使用 `gh`（GitHub CLI）
- 如果 `gh` 不可用，尝试 `glab`（GitLab CLI）
- 都不可用则提示用户安装

批量创建 Issue：
- 按优先级顺序创建（P0 先创建）
- 如有依赖关系，先创建被依赖的 Issue，拿到 issue number 后填入依赖方的描述中
- 为每个 Issue 设置对应的 label（如 label 不存在则先创建）

### 7. 报告完成

输出：
- 每个 Issue 的 URL 列表
- 按优先级分组的汇总
- 依赖关系图（如有）
- 总计创建了多少个 Issue
