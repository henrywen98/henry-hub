# 仓库管理改造 — Issue 拆解汇总

## 需求来源

会议纪要：仓库管理改造讨论

## 素材解析

### 业务目标
改造现有仓库管理模块，补齐入库审批、库存预警、批量出库、扫码盘点四大能力缺口，提升仓储运营效率并降低断货风险。

### 涉及角色
- 仓库操作员（日常出入库、盘点执行）
- 仓库主管/经理（审批入库单、设置预警规则、查看盘点报告）
- 系统管理员（配置预警参数）

### 核心功能点（共 6 个 Issue）

| 序号 | 优先级 | 标题 | 标签 | 一句话描述 |
|------|--------|------|------|-----------|
| 01 | P0 | 入库单审批流程 | feature | 入库单提交后需经主管审批才能生效入库 |
| 02 | P0 | 库存预警规则配置 | feature | 为产品设置安全库存阈值并支持自定义预警规则 |
| 03 | P0 | 库存不足自动预警通知 | feature | 库存低于安全阈值时自动发送站内预警通知 |
| 04 | P1 | 出库单批量操作 | enhancement | 支持在出库列表中批量选择并一次性提交出库 |
| 05 | P1 | 库存盘点单管理 | feature | 创建盘点任务并记录盘点结果，生成盘盈盘亏报告 |
| 06 | P1 | 扫码盘点 | feature | 支持使用扫码设备快速录入盘点数据 |

### 依赖关系

```
02 库存预警规则配置
  └──→ 03 库存不足自动预警通知（依赖 02 提供的预警阈值）

05 库存盘点单管理
  └──→ 06 扫码盘点（依赖 05 提供的盘点单基础能力）
```

- 01（入库审批）与 04（批量出库）相互独立，可并行开发
- 02 + 03 为一组，03 强依赖 02
- 05 + 06 为一组，06 强依赖 05

## 模糊点识别与假设

在分析会议纪要时，按 6 个维度扫描后发现以下模糊点。由于本次为测试运行无法与用户交互，以下列出每个模糊点及所做的合理假设：

### Q1: 入库审批流程的审批层级和角色（维度：用户角色 + 业务边界）
- **模糊点**：会议纪要仅提到"入库单没有审批流程"，未说明审批是单级还是多级、审批人角色是什么。
- **假设**：采用单级审批，由仓库主管审批即可。理由：该系统是中小型外贸 ERP，流程不宜过重，单级审批已能满足管控需求。

### Q2: 库存预警的阈值设置粒度（维度：功能范围 + 数据流向）
- **模糊点**：会议纪要说"库存预警功能缺失"，但未说明预警阈值是按产品设置还是按产品规格设置，也未说明是否需要不同类型的预警。
- **假设**：按产品规格（product_spec）粒度设置安全库存阈值，因为当前系统库存批次已按 product_spec_id 管理。预警类型先支持"低于安全库存"一种。

### Q3: 库存预警的通知方式（维度：功能范围 + 约束条件）
- **模糊点**：未说明预警触发后如何通知用户（站内消息、邮件、短信等）。
- **假设**：先支持站内通知（系统内消息提醒），后续可扩展邮件等渠道。理由：站内通知实现成本最低，且用户日常使用系统时即可看到。

### Q4: 出库批量操作的具体含义（维度：功能范围）
- **模糊点**："出库单需要支持批量操作"含义不明确——是批量创建出库单、批量审批、还是批量选择库存批次出库？
- **假设**：指在出库/发货列表中勾选多条待出库记录后一次性确认出库，减少逐条操作的重复劳动。

### Q5: 扫码盘点的设备和技术方案（维度：约束条件）
- **模糊点**：希望"扫码盘点"，但未说明使用什么扫码设备（手持扫码枪、手机摄像头、PDA等），也未说明条码类型。
- **假设**：支持通用条码扫码枪（USB HID 模式，相当于键盘输入），扫描产品上的条形码/二维码后自动识别产品并填入盘点数据。不要求移动端 APP。

### Q6: 盘点流程的审批和库存调整（维度：业务边界 + 异常场景）
- **模糊点**：会议纪要仅说"希望能扫码盘点"，未说明盘点结果发现盘盈盘亏后是否需要审批才能调整库存。
- **假设**：盘点完成后生成盘盈盘亏报告，需经主管审批确认后才自动调整库存数量。理由：库存调整涉及财务，需要管控。
