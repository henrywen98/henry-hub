# 客户对账与在线付款功能 — Issue 拆解总览

## 原始需求

> 我想给系统加一个客户对账功能，客户能看到自己的欠款情况，然后可以在线付款

## 素材解析

### 业务目标
为外贸进销存系统新增面向客户的对账和付款能力，让客户（买方）能够自助查看其所有订单的应收/已收/欠款明细，并支持在线发起付款（银行转账凭证上传），减少业务员手动催款和对账的人力成本。

### 涉及角色
- **客户（外部用户）**：查看对账单、发起付款
- **业务员 / 销售（内部用户）**：生成对账单、审核客户提交的付款凭证
- **CEO / 财务管理者（内部用户）**：查看全局应收报表、审批大额付款

### 核心功能点（初步识别）
1. 客户对账单生成与查看
2. 客户门户（外部访问入口）
3. 客户在线提交付款凭证
4. 内部审核客户付款凭证
5. 应收账款汇总报表

## 模糊点识别 & 假设

根据 clarify-dimensions 6 个维度扫描，原始需求存在以下模糊点：

### 第一轮澄清

**Q1: "客户"是否需要独立登录系统？（业务边界 & 用户角色）**
原始需求说"客户能看到自己的欠款情况"，但当前系统只有内部用户角色（CEO、供应链、销售、综管等），没有外部客户角色。需要确认：
- A) 客户通过独立的客户门户登录查看（需要新建客户账号体系）
- B) 通过分享链接/令牌方式免登录查看对账单（推荐，实现成本低，外贸客户常用）
- C) 由业务员导出对账单 PDF/Excel 发送给客户

> **假设选择 B**：通过生成带安全令牌的链接分享给客户，客户点击链接即可查看对账单并提交付款凭证。理由：外贸场景下客户分散在全球各地，注册账号门槛高；链接分享方式在外贸行业常见且实现成本可控。同时支持 C（导出 PDF/Excel）作为辅助方式。

**Q2: "在线付款"具体是什么含义？（功能范围）**
外贸场景下"在线付款"可能指：
- A) 接入第三方支付网关（PayPal、Stripe 等）进行真实线上收款
- B) 客户在线上传银行转账凭证（水单），内部人员审核确认（推荐，符合外贸 T/T 付款习惯）
- C) 仅展示付款指引（银行账户信息），客户线下转账后由内部人员手动录入

> **假设选择 B**：客户通过对账页面上传银行转账凭证（水单），系统记录后由内部人员审核确认。理由：外贸 B2B 交易主要通过 T/T 银行电汇，接入第三方支付网关的需求较低且成本高；上传水单 + 内部审核的方式最符合实际业务流程。

**Q3: 对账单的维度是什么？（数据流向）**
- A) 按客户维度汇总（一个客户所有订单的合计欠款）
- B) 按订单维度明细（每笔订单的应收/已收/欠款）（推荐）
- C) 两者都要：先看客户汇总，点击后看订单明细

> **假设选择 C**：提供客户级汇总 + 订单级明细的两层结构。理由：客户需要快速了解总欠款额，也需要核对每笔订单的收款情况。

### 第二轮澄清

**Q4: 对账单是否需要包含历史已结清的订单？（业务边界）**
- A) 仅展示有未结余额的订单（推荐，对账重点关注欠款）
- B) 展示所有订单（含已结清），可通过筛选切换

> **假设选择 B**：默认展示未结清订单，支持筛选查看所有历史订单。理由：客户可能需要回查历史交易记录。

**Q5: 客户提交付款凭证后，对已有的收款流程有什么影响？（数据流向）**
- A) 直接创建收款记录（信任客户，自动核销）
- B) 创建待审核的收款记录，审核通过后才算正式收款（推荐）
- C) 仅作为参考信息，收款仍需手动录入

> **假设选择 B**：客户提交的付款凭证创建为"待审核"状态的收款记录，经内部人员审核确认后正式入账并触发核销。理由：需要内部风控把关，防止虚假凭证。

**Q6: 分享链接的安全策略？（约束条件 - 安全性）**
- A) 链接永久有效
- B) 链接有过期时间（推荐，如 7 天或 30 天）
- C) 链接有过期时间 + 访问密码

> **假设选择 B**：链接默认有效期 30 天，过期后需业务员重新生成。理由：平衡安全性和便利性，30 天足够客户查看和处理付款。

## Issue 拆解方案

共 **6 个 Issue**，按依赖关系和优先级排序：

| 序号 | 优先级 | 标题 | 一句话描述 | 标签 |
|------|--------|------|-----------|------|
| 01 | P0 | 客户应收账款汇总查询 | 内部用户可按客户查看应收/已收/欠款汇总 | feature |
| 02 | P0 | 客户对账单生成与分享 | 业务员可为客户生成对账单并获取安全分享链接 | feature |
| 03 | P0 | 客户对账单外部查看页面 | 客户通过分享链接查看对账单明细 | feature |
| 04 | P1 | 客户在线提交付款凭证 | 客户在对账页面上传银行水单发起付款 | feature |
| 05 | P1 | 付款凭证审核与收款确认 | 内部人员审核客户提交的付款凭证并确认入账 | feature |
| 06 | P2 | 对账单导出为 PDF/Excel | 业务员可将对账单导出为文件发送给客户 | feature |

### 依赖关系

```
01 客户应收账款汇总查询
 └──> 02 客户对账单生成与分享（依赖01的数据查询能力）
       └──> 03 客户对账单外部查看页面（依赖02生成的分享链接）
             └──> 04 客户在线提交付款凭证（依赖03的外部页面）
                   └──> 05 付款凭证审核与收款确认（依赖04提交的凭证）

02 ──> 06 对账单导出为 PDF/Excel（依赖02的对账单数据）
```

### 开发建议顺序
1. **第一批（P0 核心链路）**：01 → 02 → 03（打通对账查看的完整链路）
2. **第二批（P1 付款闭环）**：04 → 05（实现付款凭证提交和审核）
3. **第三批（P2 锦上添花）**：06（导出功能）
