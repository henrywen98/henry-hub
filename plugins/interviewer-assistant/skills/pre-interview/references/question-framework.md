# 追问题设计框架

## 核心原则：把"方法论"逼到"具体执行"

候选人倾向于用方法论话术回答（"构建了数据飞轮"、"完成了产品定义"）。你的每道题要设计成**让候选人没法只停在方法论层面**，必须讲到具体事件、具体数据、具体取舍。

## 四层递进结构

对每段重要经历，按这四层设计追问：

### 第 1 层：方法论 → 具体事件

把"做了什么类"的描述逼成"给我讲一个具体的例子"。

- ❌ "你能介绍一下你做的数据闭环吗？" → 候选人会复述简历
- ✅ "数据飞轮里，你能具体讲一个 bad case 是怎么从用户反馈一路流到模型微调的吗？" → 必须给出具体案例

### 第 2 层：具体事件 → 个人贡献

一个项目有很多人，分清楚候选人的**具体责任边界**。

- ❌ "这个项目怎么做的？"
- ✅ "转型方向是你提出的还是团队已有的？你在这个项目里具体负责哪一部分？提出了哪些被采纳的意见？"

典型信号：真实贡献会说出"我做了 A，团队其他人做了 B"。包装过的回答会一直用"我们"。

### 第 3 层：个人贡献 → 数据归因

任何简历上的百分比数字，都问三个问题：

1. **口径**：这个数据的统计范围是什么？（整个产品还是你的模块？）
2. **基线**：之前是多少？怎么测的？
3. **归因**：同期还有什么别的变化？怎么证明是你的策略带来的？

- ❌ "为什么能提升 18%？"
- ✅ "这 18% 是整个产品的次月留存，还是你负责功能的次月留存？A/B 对照组是什么样的？同期还有其他功能在变吗？"

### 第 4 层：数据归因 → 判断取舍

真正有水平的候选人，一定做过**有难度的取舍决策**。问这个问题会逼出他的判断力。

- ❌ "你觉得这个项目做得好吗？"
- ✅ "在这个项目里，有没有哪次你本来想做 A，但团队决定做 B，或者反过来？你当时怎么判断的？"
- ✅ "如果现在让你重做，你会换哪个决策？"

## 核心题统一追问框架（四项）

上面的"四层递进"讲的是**一道题的追问走向**（方法论 → 具体事件 → 个人贡献 → 数据归因 → 判断取舍）。这里的"四项框架"讲的是**每道核心经历题都必须覆盖的四个维度**——更固定、更抗话术，候选人用高层叙述滑过去的话随时回补。

对每道**核心经历题**，在主问题后附这四个追问：

1. **当时输入：** 文档量 / 用户量 / 团队配置 / 时间窗口 / 预算约束？
2. **你的产出：** PRD / 评估方案 / 标注规范 / Prompt 模板 / 权限规则 / 流程文档——挑一样**具体的**，不接受"我做了方案"这种抽象表述。
3. **你拍板过什么取舍：** 砍了什么、延期了什么、为什么？
4. **结果归因：** 基线是什么、样本多大、观察了多久、谁认可这个指标？

**使用提示：** 不要把这四项塞成四个独立追问把候选人逼死；用作 checklist——候选人的主回答如果自然覆盖了三项，第四项补一下即可。只有一项都没覆盖、全是方法论话术时，才挨个追。

## 技术类问题的深度验证

针对简历里的技术术语（RAG、知识图谱、微调、Prompt 工程等），用**分层探测**：

### 第 1 层：定义层

"能解释一下 RAG 的核心流程吗？"

**合格信号：** 流畅说出"切片 → embedding → 向量检索 → 拼回上下文 → 生成"。  
**红旗信号：** 卡顿、只能说一两个关键词、顺序乱。

### 第 2 层：选型层

"什么场景下你会选 RAG 而不是直接微调？"

**合格信号：** 能给出多个维度（数据更新频率、成本、延迟、准确率、领域深度、合规要求）。  
**红旗信号：** 只能给 1 个维度（比如"文档乱不乱"），且缺乏工程权衡。

### 第 3 层：落地层

"你具体做过的 RAG 项目里，chunk 策略怎么定的？检索召回怎么评估？怎么处理幻觉？"

**合格信号：** 能说出具体的 chunk 大小、overlap、评估指标（recall@k、MRR）、降幻觉手段。  
**红旗信号：** 含糊其辞，或用方法论话术绕过具体问题。

### 第 4 层：模型 / 技术切换（针对中高级 AI PM）

真正在生产环境做过模型切换（千问 → DeepSeek、GPT-4 → Claude 等）的人，能给出**前后对比数字**。

"你们切换过程里前后的延迟、成本、拒答率、幻觉率变化大概是多少？有没有哪个场景切完反而变差？"

**合格信号：**
- 前后对比有**具体数字**（哪怕是粗略估计）
- 至少能说出**一个"切完反而变差"的场景**，并讲清怎么处理的
- 能列出切换时改的 Prompt / workflow / fallback / 回滚阈值

**红旗信号：**
- 全程都是"切换顺利 / 效果更好"
- 讲不出任何变差场景——说明要么没真正上过生产，要么评估体系不完整

### 分层判断

- 能过第 1 层 = 知道概念
- 能过第 2 层 = 有产品感
- 能过第 3 层 = 真做过
- 能过第 4 层 = 真上过生产

应届生 / 转行者过第 1-2 层就算合格。中高级 AI PM 必须过第 3 层；声称做过模型切换 / 大规模上线的，第 4 层是硬门槛。

## 工具边界题（校验"产品视角"到底多深）

简历里常写 "Python、LangChain / LlamaIndex、Coze、Dify（产品视角）"。"产品视角"四个字本身是诚实的自我限定，但边界太模糊——可能意味着方案验收，也可能意味着亲自搭 demo。得让候选人讲清楚。

挑一个简历里提到的**具体框架**问：

- "你实际做到什么层级？能改配置？搭工作流？写 demo？还是主要做方案和验收？"
- "给一个你亲手动过的具体例子——哪个项目、哪个节点、动了什么？"

**合格信号：** 边界清楚（"我能在 Coze 里搭工作流但不写自定义插件代码"）、例子具体（"给 X 项目搭了意图识别 + 补槽的工作流 demo"）。
**红旗信号：** 边界含糊（"都能做一些"），或例子抽象（"参与了搭建"）—— 说明工具那一栏是贴标签。

## 开场 & 收尾的题

### 开场（3 分钟）

- 自我介绍
- **为什么想做这个岗位**（测动机 + 对岗位的理解）
- 背景跨度类问题（跨专业 / 跨行业的话问契机）

### 收尾（5 分钟）

- **方案被否的经历**（测反思能力）
- **跨部门推动最困难的事**（测软实力）
- **你有什么问题想问我**（测对公司业务的理解深度，**这是识别"只是投简历"和"想来工作"的最强信号**）

## 一个模板（可以套用）

```markdown
### 公司 X 实习经历（5 分钟）

1. 你在这段实习里最自豪的产出是什么？为什么？
2. 你简历里写了"XXX 方法论"，能给我讲一个具体的例子吗？（第 1 层）
3. 这个方案是你提出的还是团队已有的？你具体负责哪一部分？（第 2 层）
4. 简历里的"XX% 提升"——口径是什么？对照组？归因逻辑？（第 3 层）
5. 这段经历里有没有做过哪个取舍决策？为什么那么选？（第 4 层）

> **面试官备忘：** [列出 2-3 个这段经历的简历红旗点，提醒对照候选人回答]
```
