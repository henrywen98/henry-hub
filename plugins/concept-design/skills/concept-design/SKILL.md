# 概要设计说明书生成器

根据项目中已有的 PRD 和技术架构设计文档，自动生成一份面向甲方交付的概要设计说明书。文档采用固定的7章模板结构，以功能模块设计为核心（占60%+），辅以技术选型、系统集成、安全设计等章节。

## 为什么用固定模板

概要设计说明书是交付给甲方的正式文档，需要章节完整、风格统一、篇幅充实。固定模板确保每个项目产出的概设结构一致，甲方评审时不需要适应不同的文档风格。模板来源于实际交付过的概设样例，已验证可通过甲方评审。

## 工作流程

```
Phase 1: 环境扫描 → 识别项目中的输入文档
Phase 2: 输入确认 → 确认文档 + 收集项目元信息
Phase 3: 并行提取 → 多 Agent 并行读取各文档提取结构化内容
Phase 4: 文档生成 → 按模板合成概要设计说明书
```

---

## Phase 1: 环境扫描

使用 Glob 工具扫描项目根目录下的 `.md` 文件，按以下规则识别输入文档：

### 必需文档（缺任何一个则阻断，提示用户先完成）

| 文档 | 识别规则 | 用途 |
|------|---------|------|
| PRD / 产品需求文档 | 文件名含 `PRD`/`prd`/`需求`/`产品需求` | 第3章功能模块设计的唯一来源 |
| 技术架构设计 | 文件名含 `技术架构`/`架构设计`/`architecture` | 第2/4/5/7章的来源 |

### 可选文档（缺失时降级处理，告知用户但不阻断）

| 文档 | 识别规则 | 降级策略 |
|------|---------|---------|
| 数据库设计 | 文件名含 `数据库`/`database` | 编号规则从 PRD 中提取 |
| 接口设计 | 文件名含 `接口`/`api`/`interface` | 第6章仅写概述性描述 |

交互原型（prototype 目录）有就用，不做缺失提示。若存在原型文件，Phase 4 第5章界面设计可引用原型截图描述界面风格。

### 模块数量检测

用 Grep 在 PRD 中搜索模块级标题（如二级/三级标题中含"管理"/"模块"的模式），计算模块总数 N，用于 Phase 3 决定 Agent 切分策略。

---

## Phase 2: 输入确认

向用户展示扫描结果并交互确认：

1. **文档确认**：列出识别到的文档路径，请用户确认是否正确。如有遗漏，用户可手动指定路径。

2. **齐全性检查**：
   - 必需文档缺失 → 提示：`"概设生成需要以下文档，当前缺失：XXX。请先完成这些文档再生成概设。"` → 终止
   - 可选文档缺失 → 提示：`"以下文档缺失，相关章节将使用降级策略：XXX。是否继续？"`

3. **收集项目元信息**（带默认值，用户可直接回车确认）：
   - 编制公司名称（如：XX科技有限公司）
   - 项目全称（如：XX集团 办公管理系统）
   - 文档日期（默认当天）
   - 文档版本（默认 V1.0）

4. **外部接口/数据源名称确认**：
   - 扫描 PRD 中提到的外部系统或 API（如短信服务、地图 API、第三方数据查询、统一认证等）
   - 向用户确认概设中应使用的正式名称（PRD 中的简称可能与正式接口名称不同）
   - 记录确认结果，Phase 4 生成时全文统一使用

---

## Phase 3: 并行提取

使用 Agent 工具（`subagent_type: Explore`）并行启动多个子 Agent。所有 Agent 在同一条消息中发出以实现并行。每个 Agent 返回结构化文本，由主流程在 Phase 4 中合成。

### Agent 切分策略

根据 Phase 1 检测到的模块数 N：
- N ≤ 3：1 个模块 Agent（处理全部模块）
- 4 ≤ N ≤ 10：2 个模块 Agent（各处理约 N/2 个模块）
- 11 ≤ N ≤ 20：3 个模块 Agent（各处理约 N/3 个模块）
- N > 20：ceil(N/6) 个模块 Agent，上限 5 个

每个 Agent 的 prompt 中指定负责的模块名列表。

### Agent 定义

**Agent 1..K — PRD 功能模块提取**（`subagent_type: Explore`，K 由上述策略决定）

```
读取文件 {prd_path}，提取以下模块的完整功能描述：{module_list}

对每个模块输出：
1. 模块简述（2-3句话：模块职责、核心场景、使用角色）
2. 功能设计表格，每个子功能一行：

| 序号 | 业务模块 | 功能名称 | 功能要求 | 功能设计 |

"功能设计"列按此公式填写：
操作入口 → 字段填写（列举关键字段，标注自动带出/必填/可选）
→ 保存行为 → 提交行为 → 审批/流程触发 → 后续操作 → 导出/打印

要求：
- 粒度为用户操作步骤级，模拟用户操作路径
- 必须包含：自动带出字段、保存/提交行为差异、审批触发点
- 不写技术实现（不提 API、数据库表、组件名）
- 第三人称客观描述（"用户点击…"）
- 捕获所有字段、所有业务规则、所有编号规则
```

**Agent K+1 — PRD 通用规则与汇总**（`subagent_type: Explore`）

```
读取文件 {prd_path}，提取：
1. 通用规则（适用于所有模块的规则）
2. 用户角色定义及权限层级
3. 审批流程定义（哪些功能需要审批、审批特殊说明）
4. 功能权限矩阵（各角色对各模块的权限）
5. 数据字典 / 码值（枚举值列表）
6. 编号规则汇总（各模块的编号前缀和格式）
7. 模块组间差异分析（仅当 PRD 存在多个并列的顶级模块组时执行，如不同产品线、不同业务板块共享部分子模块；若 PRD 是单一模块组结构则跳过此项）：
   - 逐一列出 PRD 中每个顶级模块组的完整子模块目录
   - 对比哪些子模块是共享的、哪些是某组独有的、哪些是某组缺失的
   - 逐一列出共享模块中业务逻辑有差异的点（如不同的审批流程）
   - 输出格式：共享模块列表 + 差异表（模块名/组A行为/组B行为）
```

**Agent K+2 — 技术架构提取**（`subagent_type: Explore`）

```
读取文件 {arch_path}，提取：
1. 技术选型：硬件环境表格 + 体系结构概述（保持概述级别，不展开分层细节）
2. 系统架构设计：整体架构图/描述、模块划分列表
3. 部署架构：网络拓扑、服务器配置、部署方案
4. 技术路线：前后端技术栈表格
5. 性能设计：性能目标、优化策略概要
6. 安全设计：安全层次、认证授权、数据加密、输入安全、审计日志、数据库备份策略
7. 非功能性需求：安全性/稳定性/可用性/可维护性需求
```

**Agent K+3（可选）— 接口设计提取**（`subagent_type: Explore`）

仅在接口设计文档存在时启动：

```
读取文件 {api_path} 的前 100 行，提取：
1. 接口规范（基础URL、协议、格式、认证方式）
2. 统一响应格式
3. 错误码定义
4. 接口模块分类列表
```

### 大文件处理

PRD 和技术架构文档通常体量较大（PRD 可达 2000+ 行，技术架构 1500+ 行）。为避免单个 Agent 上下文溢出：

- **模块 Agent**：每个 Agent 的 prompt 中指定负责的模块名列表，Agent 使用 Grep 先定位模块在文件中的行号范围，再用 Read 工具的 offset/limit 参数只读取对应区间，避免一次性加载整个文件
- **通用规则 Agent**：PRD 的通用规则、权限矩阵、审批定义、数据字典通常集中在文档首尾部分，Agent 应先 Grep 定位章节位置再分段读取
- **技术架构 Agent**：按章节名 Grep 定位后分段读取，不要一次性加载全文

### 错误处理

如果某个 Agent 超时或失败，记录警告并继续。缺失 Agent 对应的章节在 Phase 4 中标注 `<!-- TODO: 需补充 -->`。

---

## Phase 4: 文档生成

由主流程执行（非子 Agent），将所有 Agent 返回内容按模板合成。

读取模板参考文件了解完整的章节骨架和写作规范：
`${CLAUDE_PLUGIN_ROOT}/skills/concept-design/references/template.md`

### 组装步骤

1. **第1章 引言**：使用模板中的固定文本，填入用户提供的元信息（公司名/项目名/日期/版本），从输入文档名生成参考资料表
2. **第2章 技术选型**：使用技术架构 Agent 返回的硬件环境和体系结构内容
3. **第3章 功能模块设计**（核心）：
   - 3.0 系统功能总览：使用通用规则 Agent 的返回内容，生成模块全景图（ASCII art）和审批状态机（Mermaid stateDiagram-v2）
   - 3.x 各模块：按模块顺序拼接功能模块 Agent 的返回内容，为涉及审批的模块生成 Mermaid flowchart TD 流程图
   - 共享模块差异处理（仅当存在多模块组共享模块时）：在一处详写后另一处仅写差异点，差异声明必须逐模块核对 PRD（不能粗略说"复用其余模块"）
   - 汇总表：使用通用规则 Agent 返回的权限矩阵、编号规则、审批需求
4. **第4章 非业务功能需求**：使用技术架构 Agent 返回的非功能需求内容
5. **第5章 系统集成方案**：使用技术架构 Agent 返回的架构、部署、技术路线内容。**去重规则**：第5章与第2章内容会有重叠（都涉及架构和部署），处理方式为——第2章写架构概述和模块划分，第5章侧重网络拓扑、详细技术栈版本、日志方案等补充内容。重复的部分（如环境规划表、Nginx 配置表）在第5章中用"详见第 2.x 节"引用，不要复制粘贴
6. **第6章 接口设计**：使用接口 Agent 返回内容（无则从技术架构中降级为概述）
7. **第7章 安全性设计**：使用技术架构 Agent 返回的安全设计内容

### 输出

- 文件名：`概要设计说明书.md`，写入项目根目录
- 如果文件已存在，提示用户确认是否覆盖
- 使用 Write 工具一次性写入

### 输出写入策略

概设文档可能较长（800-2000+ 行）。使用 Write 工具一次性写入完整文件。如果内容确实超出单次 Write 能力，按章节分段写入：先 Write 第1-3章，再用 Edit 在文件末尾追加第4-7章。

---

## Phase 4.5: 交叉核验

文档生成后、交付用户前，执行自动验证。核验项和执行方法详见：
`${CLAUDE_PLUGIN_ROOT}/skills/concept-design/references/verification-checklist.md`

如果路径变量不可用，使用 Glob 搜索 `**/references/verification-checklist.md` 定位文件。

核验覆盖：模块覆盖率、章节占比、一致性、名称对齐、共享模块声明、copy-paste 检测、第2/5章去重。如有问题，主动修复后再交付。

---

### 生成后

完成核验后向用户报告：
- 文档统计（总行数、预估页数）
- 各章节行数占比
- 核验结果摘要
- 提示用户审阅全文

## 质量标准

生成的概要设计说明书应满足：

| 标准 | 要求 |
|------|------|
| 篇幅 | 功能模块设计章节占全文 60% 以上 |
| 独立性 | 不引用"详见PRD第X章"等跳转，所有信息自包含 |
| 一致性 | 模块名称、功能名称、字段名称与 PRD 完全对齐 |
| 覆盖率 | PRD 中每个功能页面在概设中都有对应行 |
| 审批流程 | 涉及审批的功能明确说明审批触发点 |
| 自动字段 | 每个表单说明哪些字段是系统自动填充的 |
| 保存/提交 | 每个表单区分保存（可编辑）和提交（不可编辑）的行为 |
| 流程图 | 涉及审批的模块至少1张 Mermaid 流程图 |
| 语气 | 第三人称客观描述，不出现技术实现细节 |
