# 概要设计说明书组装器

概要设计说明书是合同要求的交付物，用于方案确认阶段让客户理解"我们打算怎么做"并签字确认。本 skill 将当前目录下的素材组装为标准 7 章概设文档。

## 为什么是"组装"不是"生成"

概设在需求确认单之后产出，功能范围已经确认过。概设的真正增量是把已确认的需求包装进标准技术方案外壳。大部分技术/安全/部署章节在项目间变化不大，可以从过往概设直接复用。

所以 skill 的工作是：**把需求确认单的功能列表转成概设格式 + 从过往概设复用其他章节 + 微调项目细节**。

## 输入素材（全部在当前目录下）

```
当前目录/
├── 需求确认单 / PRD           ← 必须有（第3章功能模块的来源）
├── 过往概设 / 概设模板         ← 最重要的参考（其他所有章节的骨架和内容来源）
├── 技术架构文档               ← 可选（有则用来细化第2/5章，没有就从过往概设复用）
└── 原型 / prototype           ← 可选（有则辅助第5.5章界面设计描述）
```

---

## 工作流程

```
Phase 1: 素材扫描 → 识别当前目录下的输入文件
Phase 2: 确认 → 确认文件 + 收集项目元信息
Phase 3: 内容提取 → 从素材中提取结构化内容
Phase 4: 组装输出 → 按模板合成概要设计说明书
Phase 5: 核验报告 → 自动验证 + 交付用户
```

---

## Phase 1: 素材扫描

使用 Glob 扫描当前目录下的文档文件（`.md`、`.docx`、`.doc`、`.pdf`），按以下规则识别：

### 必需素材

| 素材 | 识别规则 | 用途 |
|------|---------|------|
| 需求确认单 / PRD | 文件名含 `需求确认`/`确认单`/`PRD`/`prd`/`需求` | 第3章功能模块的来源 |

缺失则阻断，提示用户先准备需求文档。

### 关键参考

| 素材 | 识别规则 | 说明 |
|------|---------|------|
| 过往概设 / 概设模板 | 文件名含 `概设`/`概要设计`/`HLD`/`模板` | 第1/2/4/5/6/7章的骨架来源 |

缺失时提示用户：过往概设是最重要的参考素材，缺少它技术/安全等章节会使用内置模板（质量和匹配度会降低）。用户确认后继续。

### 可选素材

| 素材 | 识别规则 | 降级策略 |
|------|---------|---------|
| 技术架构文档 | 文件名含 `技术架构`/`架构设计`/`architecture` | 无则从过往概设复用技术章节 |
| 接口设计文档 | 文件名含 `接口`/`api`/`interface` | 无则第6章仅写概述 |
| 原型文件 | `prototype`/`原型` 目录或文件 | 无则第5.5章用通用描述 |

---

## Phase 2: 确认

1. **素材确认**：列出识别到的文件路径，请用户确认。用户可手动指定路径或补充遗漏文件。

2. **收集项目元信息**（带默认值）：
   - 项目全称
   - 项目简称（文档内引用）
   - 承建单位名称
   - 文档日期（默认当月）
   - 文档版本（默认 V1.0）

3. **快速确认**：问用户"技术架构和过往概设基本一致，还是有需要调整的地方？"如果一致，技术章节直接复用；如果有变化，记录变化点。

---

## Phase 3: 内容提取

使用 Agent 工具（`subagent_type: Explore`）并行提取。所有 Agent 在同一条消息中发出。

### Agent 切分策略

根据需求文档中的模块数 N（用 Grep 搜索模块级标题计算）：
- N ≤ 5：1 个模块 Agent
- 6 ≤ N ≤ 15：2 个模块 Agent
- N > 15：3 个模块 Agent，每个处理约 N/3

### Agent 定义

**Agent 1..K — 需求功能提取**（`subagent_type: Explore`）

```
读取文件 {requirements_path}，提取以下模块的功能描述：{module_list}

对每个模块输出：
1. 模块简述（2-3句话：模块职责、核心场景、使用角色）
2. 功能设计表格，每个子功能一行：

| 序号 | 业务模块 | 功能名称 | 功能要求 |

"功能要求"列写作规范：
- 以"用于{业务场景描述}"开头
- 多子功能用"功能包括但不限于："引出
- 每个子功能以"支持"开头，分号分隔
- 量化用"不少于X个"，举例用"包括但不限于：{示例}等"
- 简单功能可只有用途说明

要求：
- 完整覆盖需求文档中该模块的所有功能点
- 第三人称客观描述
- 不写技术实现细节
```

**Agent K+1 — 过往概设提取**（`subagent_type: Explore`）

```
读取文件 {past_hld_path}，提取以下内容：

1. 第1章引言：编写目的、读者对象的完整文本
2. 第2章技术选型：硬件环境表格 + 体系结构描述
3. 第4章非业务功能需求：安全性需求 + 稳定性需求的完整文本
4. 第5章系统集成方案：架构设计、网络拓扑、部署设计、技术路线、界面设计的完整文本
5. 第6章接口设计：与外部系统接口 + 内部接口的完整文本
6. 第7章安全性设计：7.1-7.7 所有子节的完整文本

对每个章节，原样保留文本内容，仅标注需要替换项目名称的位置。
```

**Agent K+2（可选）— 技术架构补充**（`subagent_type: Explore`）

仅在有技术架构文档且用户表示架构有变化时启动：

```
读取文件 {arch_path}，提取与过往概设不同的内容：
1. 硬件环境变化（服务器数量/配置调整）
2. 技术栈变化（新增或替换的技术组件）
3. 部署方案变化
4. 安全设计变化
```

### 大文件处理

需求文档和过往概设通常较大。每个 Agent 应使用 Grep 先定位章节位置，再用 Read 的 offset/limit 参数分段读取，避免一次性加载全文。

## Phase 4: 组装输出

由主流程执行，将所有 Agent 返回内容按模板合成。

读取模板参考文件了解完整的章节骨架和写作规范：
`${CLAUDE_PLUGIN_ROOT}/skills/concept-design/references/template.md`
如果路径变量不可用，使用 Glob 搜索 `**/concept-design/**/template.md` 定位文件。

### 组装逻辑

概设的 7 章按来源分为两类：

**从过往概设复用（微调项目细节）：**

| 章节 | 组装方式 |
|------|----------|
| 第1章 引言 | 复用过往概设文本，替换项目名称/日期/承建单位 |
| 第2章 技术选型 | 复用过往概设，如有技术架构变化则合并更新 |
| 第4章 非业务功能需求 | 复用过往概设文本，替换业务场景示例 |
| 第5章 系统集成方案 | 复用过往概设，如有架构变化则合并更新 |
| 第7章 安全性设计 | 直接复用，几乎不改 |

如无过往概设，使用 `references/template.md` 中的内置模板（含完整的固定模板文本和样本摘录）。如连技术架构文档也没有，则向用户交互收集：技术架构选型（前后端/数据库）、硬件清单（服务器角色和数量）、部署方案概述。

**从需求确认单转换（核心工作）：**

| 章节 | 组装方式 |
|------|----------|
| 第3章 功能模块设计 | 需求 Agent 返回的模块简述 + 功能表格，按子系统编排为 3.2.x 结构 |
| 第6章 接口设计 | 从需求文档中提取的外部系统对接需求 + 过往概设的接口框架 |

### 第3章组装细节

第3章是概设的核心（目标占全文 60%+）：

1. **3.1 功能结构图**：基于模块 Agent 返回的模块列表，生成功能结构树
2. **3.2.x 各子系统**：每个子系统一个三级标题，下方紧跟功能表格
3. 功能表格列固定为：序号 | 业务模块 | 功能名称 | 功能要求

### 输出

- 文件名：`概要设计说明书.md`，写入当前目录
- 如果文件已存在，提示用户确认是否覆盖
- 优先使用 Write 工具一次性写入完整文件；若单次写入失败，改为分段写入（先写第1-3章，再用 Edit 追加第4-7章）

---

## Phase 5: 核验报告

文档生成后、交付用户前，执行自动验证。核验项和执行方法详见：
`${CLAUDE_PLUGIN_ROOT}/skills/concept-design/references/verification-checklist.md`
如果路径变量不可用，使用 Glob 搜索 `**/concept-design/**/verification-checklist.md` 定位文件。

完成核验后向用户报告：
- 文档统计（总行数、预估页数）
- 各章节行数占比
- 核验结果摘要
- 提示用户审阅全文（概设是合规交付物，人工审阅是必要步骤）

## 写作风格约束

| 维度 | 规范 |
|------|------|
| 人称 | 第一人称复数（"我们"）或无主语被动句 |
| 语气 | 正式、专业、客观 |
| 句式 | 短句为主，避免长复合句 |
| 术语 | 行业标准术语，首次出现缩写需给全称 |
| 编号 | 阿拉伯数字，最多三级（如 3.2.1） |
| 表格 | 首行为表头 |
| 引用 | 书名号《》包裹完整文件名 |

## 质量标准

| 标准 | 要求 |
|------|------|
| 篇幅 | 功能模块设计章节占全文 60% 以上 |
| 独立性 | 不引用"详见PRD第X章"等跳转，所有信息自包含 |
| 一致性 | 模块名称、功能名称与需求确认单完全对齐 |
| 覆盖率 | 需求确认单中每个功能点在概设中都有对应行 |
| 语气 | 第三人称客观描述，不出现技术实现细节 |
